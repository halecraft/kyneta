# Plan 006.1: Unified Weighted Datalog Evaluator

## Background

Plan 006 delivered an end-to-end incremental pipeline. The Datalog evaluator
was made incremental via two strategies: semi-naive delta propagation for
monotone strata (no retractions), and DRed (delete-and-rederive) for negation
strata and any stratum when retractions are present. DRed wipes all derived
facts for a stratum and recomputes from scratch — O(|stratum|) per step rather
than O(|Δ|). A snapshot-and-diff at the boundary recovers the output delta.

This was a pragmatic choice. The batch evaluator had no weight infrastructure,
provenance tracking was complex, and DRed's cost was bounded by slot/parent
count for the default rules. But it left the Datalog evaluator as the only
stage in the pipeline that falls back to recomputation rather than computing
deltas from deltas. It also left two separate evaluator implementations
(batch in `evaluate.ts`, incremental in `incremental-evaluate.ts`) sharing
inner functions but duplicating stratum-level logic in three places.

DBSP (Budiu & McSherry, 2023) provides the principled alternative: Z-set
weights on relations. A derived fact's weight is the product of its input
weights. When an input is retracted (weight −1), the retraction propagates
through joins automatically. `distinct` (clamp to 0/1) after each semi-naive
iteration prevents weight explosion from transitive closure. The same
algorithm handles monotone and negation strata — no DRed, no snapshot-and-diff,
no special cases.

## Problem Statement

1. **Two evaluators.** `evaluate.ts` (batch, stateless) and
   `incremental-evaluate.ts` (incremental, stateful) implement the same
   algorithm with duplicated stratum-level logic. `fullRecompute()` in the
   incremental evaluator is a copy-paste of the batch evaluator. Bug fixes
   or enhancements must be applied in multiple places.

2. **DRed is not incremental.** For negation strata, the evaluator wipes all
   derived facts and recomputes from scratch. For monotone strata with
   retractions, it does the same. This is O(|stratum|) per step, not O(|Δ|).

3. **Snapshot-and-diff is redundant work.** Before each step, the evaluator
   clones all derived predicates. After the step, it diffs against the clone.
   This is O(|derived facts|) per step — work that exists solely because the
   evaluator doesn't produce deltas natively.

4. **`evaluateMonotoneStratumIncremental` ignores its `_inputDelta`
   parameter.** The initial pass evaluates all rules against the full DB
   rather than seeding semi-naive from the delta. The "incremental" monotone
   path is O(|DB|²), not O(|Δ|×|DB|).

5. **Dead code.** `isReferencedByStratum`, `diffFuguePairs`, `flattenPairs`
   are defined but never called. `snapshotDerivedPredicates` and
   `diffDatabases` exist only to support the snapshot-and-diff pattern.

## Success Criteria

1. One evaluator implementation. `evaluate(rules, facts)` is a convenience
   wrapper over `createEvaluator(rules).step(allFactsAsPositive)`. No
   duplicated stratum-level logic.

2. `Relation` and `Database` support integer weights per fact. Ground facts
   enter with weight ±1. Derived facts accumulate weights from joins.
   `distinct` clamps to 0/1 after each semi-naive iteration.

3. The semi-naive loop propagates both +1 and −1 deltas through joins and
   negation. No DRed. No snapshot-and-diff. The output delta falls out of
   the Z-set arithmetic.

4. Monotone strata are seeded from the input delta (the `_inputDelta` fix).
   Negation strata use the same weighted semi-naive — the algorithm is
   uniform.

5. All 1198 existing tests pass. The batch pipeline (`solve()`) continues to
   serve as the differential testing oracle — it calls through to the same
   unified evaluator.

6. Dead code removed (see Migration Inventory below for the full list).

## Gap Analysis

### What Exists

- **`Relation`** — `Map<string, FactTuple>`. Boolean presence. `add`,
  `has`, `remove`, `tuples`, `union`, `difference`, `clone`.
- **`Database`** — `Map<string, Relation>`. `addFact`, `removeFact`,
  `hasFact`, `mergeFrom`, `clone`, `size`.
- **`Substitution`** — `ReadonlyMap<string, Value>`. No weight. Used
  throughout `unify.ts`, `evaluate.ts`, `aggregate.ts`.
- **`evaluateRule`** / **`evaluateRuleSemiNaive`** — take `Database`
  params, return `Fact[]`. No weights.
- **`evaluatePositiveAtom`** — iterates `relation.tuples()`, calls
  `matchAtomAgainstRelation`. Returns `Substitution[]`.
- **`evaluateNegation`** — filters substitutions where negated atom has
  no match. Returns `Substitution[]`.
- **`groundHead`** — produces `Fact[]` from substitutions. Deduplicates.
- **`evaluate()`** — stateless batch entry point. Stratifies, evaluates
  each stratum, returns `Database`.
- **`createIncrementalDatalogEvaluator()`** — stateful. Maintains
  persistent `Database`. DRed for negation strata. Snapshot-and-diff for
  output deltas.
- **`ZSet<T>`** — `ReadonlyMap<string, ZSetEntry<T>>`. Full algebra in
  `base/zset.ts`. Used at pipeline boundaries.

### What's Missing

1. **Weighted `Relation`** — `Map<string, { tuple: FactTuple, weight: number }>`.
   `addWeighted(tuple, weight)` sums weights. `getWeight(tuple)` returns
   weight. `tuples()` returns weight > 0 tuples (backward compatible).
   Zero-weight entries pruned.
2. **Weighted `Database`** — `addWeightedFact(f, weight)`. Ground facts
   enter with weight ±1.
3. **Weighted `Substitution`** — redefine `Substitution` from
   `ReadonlyMap<string, Value>` to
   `{ bindings: ReadonlyMap<string, Value>, weight: number }`.
   One type, one name — not a new type alongside the old one. Positive
   atom join: weight = sub.weight × tuple.weight. Negation: weight
   preserved (filter). Guard: weight preserved (filter).
4. **Weighted rule evaluation** — `evaluateRule` takes and produces
   `Substitution`s (now carrying weights). `groundHead` produces
   `{ fact: Fact, weight: number }[]`. Multiple derivations of the same
   fact sum weights. No parallel unweighted variants — one set of
   functions, weights always present.
5. **Unified evaluator** — one `createEvaluator(rules)` that subsumes both
   batch and incremental paths. `step(delta)` is the general case.
   `evaluate(rules, facts)` is a convenience wrapper.
6. **Scoped `distinct`** — after each semi-naive iteration, clamp weights
   to 0/1 only for entries modified during that iteration (tracked via a
   dirty set). O(|modified|) per iteration, not O(|relation|).

## Architecture

### Weight Propagation Through the Evaluation Core

The fundamental change: `Relation` stores `{ tuple, weight }` per entry.
Rule evaluation threads weights through substitutions. The semi-naive loop
propagates weighted deltas.

`Substitution` is redefined from a raw Map to a struct:

```typescript
// Before (types.ts):
type Substitution = ReadonlyMap<string, Value>;

// After (types.ts):
interface Substitution {
  readonly bindings: ReadonlyMap<string, Value>;
  readonly weight: number;
}
```

No new type name — `Substitution` remains `Substitution`. The access
pattern changes: `sub.get('X')` → `sub.bindings.get('X')`. All evaluation
functions — in `evaluate.ts`, `unify.ts`, and `aggregate.ts` — operate on
the enriched `Substitution`. Functions that don't inspect the weight
(matching, guards) pass it through unchanged. Functions that need it
(positive atom join, groundHead) read/multiply it. `EMPTY_SUBSTITUTION`
becomes `{ bindings: new Map(), weight: 1 }`.

**Positive atom join:** When a `Substitution` matches a tuple with weight
`w`, the extended substitution's weight is `sub.weight × w`. This is the
provenance semiring product.

**Negation:** A boolean filter. If the negated predicate has a matching fact
with weight > 0 (after `distinct`), the substitution is filtered out
(dropped). Otherwise it passes through with weight preserved. Negation
does not produce fractional weights.

**Guard:** A boolean filter. Weight preserved on pass, substitution dropped
on fail.

**Aggregation:** For `count`, each contributing fact's weight matters. For
`min`/`max`/`sum`, facts with weight > 0 participate. Deferred: our default
rules don't use aggregation in the incremental path, and aggregation's
weight semantics need careful per-function treatment. Aggregation continues
to use the existing boolean semantics (facts with weight > 0 are "present").
The output substitution's weight is set to 1 (aggregation is a group-by
boundary that resets provenance).

**`groundHead`:** Produces `{ fact, weight }[]`. Multiple substitutions
yielding the same fact have their weights summed (Z-set addition on the
output relation).

**`distinct`:** Applied after each semi-naive iteration, but only to entries
that were modified during that iteration. The evaluator tracks a dirty set
(`Set<string>` of serialized tuple keys) during each iteration. After the
iteration, for each dirty key: weight > 0 → 1, weight < 0 → −1, weight 0
→ prune. This is O(|modified entries|) per iteration, not O(|relation|).
Prevents weight explosion from transitive closure (`fugue_descendant`) and
ensures the binary "present/absent" signal is preserved for negation.

### Cross-Time Semi-Naive with Weights

At outer time step t, when `Δ_ground[t]` arrives:

1. Apply `Δ_ground[t]` to the accumulated weighted `Database`:
   for each `(fact, weight)` in the delta, call
   `db.addWeightedFact(fact, weight)`.
2. For each affected stratum (bottom-up):
   a. Seed `currentDelta` from the changed input predicates (facts
      whose weight changed). This is the `_inputDelta` fix.
   b. Semi-naive iterate with weights:
      - For each rule, for each positive atom index, evaluate with that
        atom against `currentDelta`, others against full `db`. Weights
        multiply through joins.
      - `groundHead` produces weighted facts. Sum into a `nextDelta`
        relation.
      - Merge `nextDelta` into accumulated `db` (weight addition).
      - Apply `distinct` to modified entries only (dirty set).
      - New `currentDelta` = entries whose post-`distinct` weight
        differs from pre-iteration weight.
   c. Repeat until convergence (currentDelta is empty).
   d. The stratum's output delta = entries in derived predicates whose
      weight changed from the pre-step value (0→1 or 1→0).
3. Propagate stratum deltas upward to higher strata.
4. Collect output delta: derived facts whose weight changed.

This algorithm is uniform across monotone and negation strata. When a
ground fact with weight −1 arrives:
- Monotone stratum: the −1 propagates through joins, producing −1
  derivations. `distinct` clamps. Facts that lost all support reach
  weight 0 and are retracted.
- Negation stratum: the lower stratum may retract a `superseded` fact.
  The `winner` rule's `not superseded(...)` now succeeds for a
  previously-blocked substitution, producing a +1 winner. The −1 for
  the old winner and +1 for the new winner are the natural output.

### Unified Evaluator: Batch as Degenerate Case

```typescript
createEvaluator(rules) → Evaluator
  .step(deltaFacts: ZSet<Fact>) → StepResult
  .step(deltaFacts: ZSet<Fact>, deltaRules: ZSet<Rule>) → StepResult
  .currentDatabase() → Database
  .currentResolution() → { winners, fuguePairs }
  .reset() → void

evaluate(rules, facts) → Database
  // Convenience: createEvaluator(rules).step(allFactsAsPositive).db
```

The batch pipeline's `solve()` calls `evaluate(rules, facts)` which creates
a fresh evaluator and feeds all facts as +1. The incremental pipeline's
evaluation stage holds a persistent `Evaluator` and calls `step(delta)`.

The "correctness oracle" property is preserved: the batch path creates a
fresh evaluator per call (no accumulated state), so it's independent of
the incremental path's accumulated state. The *implementation* is shared,
but the *state* is separate — which is the right kind of independence for
an oracle.

### Resolution Extraction Without Snapshot-and-Diff

Currently, the incremental evaluator snapshots derived predicates before
and after each step, then diffs to find what changed. With weighted
relations, the delta falls out of the semi-naive loop directly: any
derived fact whose weight transitions between 0 and non-zero is part of
the output delta.

The evaluator tracks weight changes during the step and emits them as
`ZSet<Fact>`. The `winnerFactsToResolution` and `fuguePairFactsToResolution`
converters continue to work — they receive the derived fact delta and
convert to typed resolution deltas. The only change is that the delta is
computed from weight changes rather than from snapshot-and-diff.

`currentResolution()` reads the `winner` and `fugue_before` relations from
the accumulated Database. Since `tuples()` returns weight > 0 tuples, this
works unchanged.

### Backward Compatibility of `Relation` and `Database`

`Relation.add(tuple)` becomes sugar for `addWeighted(tuple, 1)`. Returns
true if the fact was newly present (weight went from ≤0 to >0).
`Relation.has(tuple)` checks weight > 0. `Relation.tuples()` returns
tuples with weight > 0 (backward compatible). `Relation.remove(tuple)`
becomes `addWeighted(tuple, -currentWeight)` — zeroes out the entry.

`Database.addFact(f)` calls `addWeighted(f, 1)`. `Database.hasFact(f)`
checks weight > 0. `Database.removeFact(f)` zeroes the weight.

Existing batch evaluator tests that create `Database`, `addFact`, `hasFact`,
`tuples()` — all work unchanged. The weight field is invisible to consumers
that don't ask for it.

### Functional Core / Imperative Shell

The stratum evaluation logic is a pure-ish function: given rules, a mutable
database, and an input delta, run weighted semi-naive to convergence and
return the output delta. This is independently testable:

```typescript
function evaluateStratumFromDelta(
  rules: readonly Rule[],
  db: Database,
  inputDelta: Database,
): Database  // output delta (facts whose weight changed)
```

`db` is mutated as a side effect of convergence (weight additions, `distinct`
clamping). The return value is the net delta. The `createEvaluator` closure
is the imperative shell that holds accumulated state and orchestrates
stratum evaluation across time steps.

**Note on `aggregate.ts`:** `evaluateAggregation` constructs result
substitutions via `new Map(baseSub)` — this becomes
`new Map(baseSub.bindings)`. The output substitution's weight is set to 1
(aggregation is a group-by boundary that resets provenance). This is the
one spot where the `Substitution` type change requires a structural code
change rather than a mechanical `.bindings` insertion.

## Transitive Effect Analysis

Every consumer of `Relation` is affected:
- `Database` — delegates to `Relation`. Updated in Phase 1.
- `evaluate.ts` — calls `relation.tuples()`, `db.hasFact()`,
  `db.addFact()`, `db.mergeFrom()`. All backward compatible after Phase 1.
  Weighted rule evaluation uses `relation.getWeight()` for weight
  multiplication.
- `incremental-evaluate.ts` — calls `relation.tuples()`, `db.addFact()`,
  `db.removeFact()`, `db.mergeFrom()`. Subsumed in Phase 2.
- `aggregate.ts` — calls `db.getRelation(pred).tuples()`. Backward
  compatible (`tuples()` returns weight > 0 tuples).
- `unify.ts` — operates on enriched `Substitution` after Phase 1.
  Matching functions access `sub.bindings` and pass weight through
  unchanged.
- `resolve.ts` — calls `db.getRelation(pred).tuples()`. Backward

  `Relation`. No change.

### `winnerFactsToResolution` anti-cancellation workaround

With proper weight propagation, a winner change produces a derived fact
delta where the old winner's weight goes from 1 to 0 (delta −1) and the
new winner's weight goes from 0 to 1 (delta +1). These are different
facts (different CnId, content) with different keys. Under `zsetMap`
keyed by slotId, they would still collide. The anti-cancellation
workaround (group by slotId, apply replacement semantics) remains
necessary — it's a key-collision issue at the resolution extraction layer,
not a weight-propagation issue. No change needed.

## Testing Strategy

### Three-Way Oracle

For every insertion sequence, verify:
1. `evaluate(rules, allFacts)` — batch (fresh evaluator, all facts at once)
2. `createEvaluator(rules) + step(allFacts)` — incremental, one big step
3. `createEvaluator(rules) + step(fact₁) + step(fact₂) + ...` — incremental,
   one fact per step

All three must produce the same `Database`. This replaces the old
batch-vs-incremental oracle with a stronger property.

### Weight-Specific Tests

- Weight propagation through joins (positive atom × positive atom).
- Weight propagation through negation (filter on weight > 0).
- `distinct` clamping prevents weight explosion on transitive closure.
- Retraction cascade: ground fact −1 → derived fact weight drops to 0.
- Re-derivation: retracted fact has alternative support → weight stays 1.

### Stratum Evaluation Unit Tests

`evaluateStratumFromDelta` is a pure(ish) function — testable in isolation
with small rule sets and hand-crafted deltas. Tests should cover:
- Positive-only rules with +1 delta (new derivations).
- Positive-only rules with −1 delta (retraction propagation).
- Negation rules with delta that changes negation outcome.
- Recursive rules (transitive closure) with `distinct` convergence.

### Permutation Tests

Existing permutation tests (3 values in all orderings produce same
resolution) continue to work and also validate weight consistency.

### Custom Rule Tests

Existing custom rule tests (reversed LWW, rule addition/retraction)
continue to work via the unified evaluator.

## Alternatives Considered

### Keep DRed, Only Fix `_inputDelta`

The minimal fix: use the input delta to seed semi-naive in
`evaluateMonotoneStratumIncremental`. This makes monotone strata O(|Δ|)
but leaves negation strata at O(|stratum|). Rejected because: (a) it
perpetuates the DRed/snapshot-and-diff complexity, (b) it doesn't help
negation strata, (c) Plan 008 queries may have large negation strata,
and (d) we'd still have two evaluator implementations.

### Create Parallel Weighted Function Variants

Create `evaluateRuleWeighted`, `evaluatePositiveAtomWeighted`, etc.
alongside the existing unweighted versions, with unweighted versions
becoming thin wrappers. Rejected because: (a) 10+ functions where the
weighted version IS the real implementation and the wrapper strips weights
— pure boilerplate, (b) the wrapper and real function drift if anyone
edits the wrapper, (c) redefining `Substitution` to carry a weight
eliminates the need for parallel variants entirely.

### Add Weights Only to a New `WeightedRelation` Type

Plan 006 initially considered this (see LEARNINGS.md "ZSet<Fact> Is the
Right Unification Point"). Rejected because: (a) conversion between
`Relation` and `WeightedRelation` at every boundary is pure overhead,
(b) the batch evaluator's inner functions take `Database` parameters —
a separate type requires adapters, (c) one type with an extra field is
simpler than two isomorphic types.

### Use `ZSet<Fact>` Internally Instead of Weighted `Relation`

Replace `Database` with `Map<string, ZSet<Fact>>`. Rejected in Plan 006
because the semi-naive loop mutates `db` between iterations. This
concern persists — `ZSet<T>` is `ReadonlyMap` by design (immutable for
algebraic safety). `Relation` is mutable (mutated by `mergeFrom` between
iterations). The right boundary: `ZSet` for inter-stage deltas
(immutable), weighted `Relation` for intra-evaluator state (mutable).

### Apply `distinct` to Entire Relation After Each Iteration

Walk all entries in a relation to clamp weights. O(|relation|) per
iteration. Rejected in favor of scoped `distinct` with a dirty set:
track which entries were modified during the iteration, clamp only those.
O(|modified entries|) per iteration.

### Full Provenance Tracking (Derivation DAGs per Fact)

Track which specific ground facts support each derivation, enabling
precise retraction. DBSP subsumes this via Z-set arithmetic — weights
encode multiplicity, which is sufficient for determining when a fact
loses all support. Per-fact DAGs are more expensive (memory and
bookkeeping) and unnecessary when `distinct` keeps weights at 0/1.
Rejected in favor of the lighter Z-set weight approach.

### Defer to Plan 008

Address the evaluator during query layer implementation. Rejected because:
(a) Plan 008 would inherit duplicated evaluator code and need to add
weights in two places, (b) the DRed/snapshot-and-diff complexity is
carrying cost in every future plan that touches the evaluator, (c) the
unification is independently valuable for code health.
