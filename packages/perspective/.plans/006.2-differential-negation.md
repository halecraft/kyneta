# Plan 006.2: Differential Negation and True Incremental Retraction

## Background

Plan 006.1 delivered a unified weighted Datalog evaluator based on DBSP
Z-set weight propagation. The infrastructure is solid: `Relation` stores
integer weights per fact, `Substitution` carries weights through joins
(provenance semiring product), `distinct` clamps via a scoped dirty map,
and `extractDelta` detects zero-crossings without snapshot-and-diff.

However, the evaluator falls back to **wipe-and-recompute** in two cases:

1. **Negation/aggregation strata** — always, even for insertion-only deltas.
2. **Positive strata with retractions** — whenever the input delta contains
   any −1 entry.

The wipe-and-recompute path deletes all derived facts for the stratum,
re-evaluates from scratch, and uses the dirty map to extract the delta.
This is O(|stratum|) per step rather than O(|Δ|). The `retractionsPresent`
flag propagates conservatively through strata: if stratum 0 produces −1
deltas, all higher strata also wipe-and-recompute.

The Z-set abelian group structure supports true incremental retraction
propagation — including through negation. The weight infrastructure is
80% in place. Three capabilities are missing:

1. **Dual-weight `Relation`**: The current `Relation` stores a single
   `weight` per fact, which is clamped to 0/1 by `applyDistinct`. This
   destroys the Z-set multiplicity — the count of independent derivation
   paths. When a fact has multiple derivation paths (e.g., `superseded(alice, S)`
   derived by both bob and charlie), the clamped weight is 1. Retracting
   one path (−1) drops it to 0, incorrectly retracting the fact even
   though alternative support exists. DBSP requires maintaining the true
   Z-set multiplicity (`weight`) alongside the clamped presence signal
   (`clampedWeight`).

2. **Bidirectional delta propagation**: `applyDerivedFact` only seeds the
   next semi-naive iteration for insertions (absent→present). It ignores
   retractions (present→absent), so cascading retractions don't propagate.

3. **Differential negation**: `evaluateNegation` is a boolean filter — it
   passes or blocks substitutions. It cannot produce negative-weight
   substitutions when a previously-absent negated fact appears (blocking
   an existing derivation) or when a previously-present negated fact
   disappears (unblocking a new derivation).

## Problem Statement

1. **Wipe-and-recompute is O(|stratum|) per step.** For the default rules
   with the native fast path, this is irrelevant (native solvers bypass
   Datalog). For custom Layer 2+ rules and future Layer 3+ user queries,
   negation strata may be large. Wipe-and-recompute becomes a bottleneck
   that scales with total derived facts, not with the delta size.

2. **Positive strata with retractions fall back unnecessarily.** The weight
   infrastructure already supports −1 propagation through joins. The only
   gap is that `applyDerivedFact` doesn't feed retractions into the
   semi-naive loop and `evaluatePositiveAtom` filters out negative-weight
   entries from the delta.

3. **`distinct` destroys multiplicity.** The current `applyDistinct`
   clamps the single `weight` field to 0/1. This means a fact derived
   via two paths has weight 1 in the accumulated DB. Retracting one path
   drops the weight to 0 — an incorrect retraction. This is the
   **three-value LWW bug**: alice (L=10), bob (L=20), charlie (L=30) in
   the same slot; `superseded(alice, S)` is derived by both bob and
   charlie; retracting charlie would incorrectly retract
   `superseded(alice, S)` even though bob still supersedes alice.

4. **Dead code.** `wipeAndRecompute` (~90 LOC), `evaluateNegationStratum`
   (~40 LOC, already unreachable), the `retractionsPresent` flag threading,
   the `inputHasRetractions` parameter, and `hasNegOrAgg` routing in
   `evaluateStratumFromDelta` are all artifacts of the wipe-and-recompute
   approach.

5. **The evaluator's public contract is misleading.** The test suite
   includes a section titled "retraction through negation without DRed" but
   the underlying implementation **is** DRed (wipe-and-recompute). The tests
   verify outcomes, not mechanisms.

## Success Criteria

1. `evaluateStratumFromDelta` is a single algorithm — no conditional
   routing based on `hasNegOrAgg` or `inputHasRetractions`. The same
   weighted semi-naive loop handles positive strata, negation strata,
   insertions, and retractions uniformly.

2. For positive strata with retractions: O(|Δ|×|DB|) per step, not
   O(|stratum|). Retractions cascade through recursive joins via the
   semi-naive loop.

3. For negation strata: O(|Δ_negated|×|DB_positive| + |Δ_positive|×|DB|)
   per step. Differential negation processes only the changed entries in
   the negated relation, not the entire relation.

4. Facts with multiple derivation paths survive partial retraction. The
   dual-weight scheme (`weight` for Z-set multiplicity, `clampedWeight`
   for the post-`distinct` presence signal) correctly tracks how many
   independent paths support each fact.

5. `wipeAndRecompute`, `evaluateNegationStratum`, and all conditional
   routing code are deleted.

6. All existing tests pass. The three-way oracle (batch ≡ single-step ≡
   one-at-a-time) continues to hold. Retraction tests verify outcomes
   that are now achieved via true weight propagation rather than
   wipe-and-recompute.

7. New tests verify the mechanism: intermediate weight states, cascading
   retractions through recursive rules, differential negation through
   stratified negation, and multi-path derivation survival.

8. Aggregation strata continue to work correctly. Aggregation is a
   group-by boundary that resets provenance, so its incremental story is
   different from negation. This plan retains wipe-and-recompute scoped
   to aggregation-only strata as a documented limitation.

## Gap Analysis

### What Exists

- **Weight-carrying `Substitution`**: `{ bindings, weight }` flows through
  all evaluation functions.
- **Provenance product in joins**: `evaluatePositiveAtom` multiplies
  `sub.weight × tuple.weight`.
- **Z-set addition in `groundHead`**: Duplicate facts sum weights.
- **Dirty-map `distinct`**: `applyDistinct` clamps to 0/1 on modified
  entries only.
- **Zero-crossing delta extraction**: `extractDelta` compares `preWeight`
  to current weight, emitting +1 or −1.
- **`Relation.addWeighted`**: Handles negative weights correctly (stores,
  sums, prunes at zero).
- **`Relation.allWeightedTuples`**: Returns all entries including negative
  weights (for delta databases).
- **Inter-stratum delta propagation**: `step()` already merges stratum
  output deltas into `currentInputDelta` for higher strata.

### What's Missing

1. **`Relation` stores a single `weight` that `applyDistinct` clamps.**
   After clamping, multiplicity is lost. A fact with 2 derivation paths
   has weight 1. A −1 delta drops it to 0 instead of 1. DBSP requires
   maintaining the pre-distinct weight (true Z-set multiplicity) alongside
   the post-distinct presence signal. The fix: dual weights per entry —
   `weight` (true multiplicity) and `clampedWeight` (the `distinct` output,
   always 0 or 1).

2. **`evaluatePositiveAtom` uses `weightedTuples()`** (weight > 0 only).
   When matching against a delta database, −1 entries are invisible. Must
   use `allWeightedTuples()` **for delta DB joins** so that retracted delta
   entries produce negative-weight substitutions via the provenance product.
   Cannot use `allWeightedTuples()` unconditionally — for the accumulated
   DB, entries may have `weight > 1` (true multiplicity), which would cause
   weight explosion in recursive joins. The fix: a conditional parameter.

3. **`applyDerivedFact` only propagates insertions.** When a fact crosses
   from present to absent (clampedWeight > 0 → 0), the retraction is not
   added to `nextDelta`. Cascading retractions in recursive rules stall.

4. **Semi-naive convergence and seed checks use `Database.size`** (weight
   > 0 count). A delta containing only −1 entries has `size === 0`. This
   affects both the convergence check (`currentDelta.size > 0`) and the
   seed guard (`inputDelta.size > 0`), causing premature loop termination
   and silent retraction-skipping respectively.

5. **No `evaluateDifferentialNegation` function.** When the negated
   relation changes, existing bindings from positive atoms must be
   re-evaluated: newly-present negated facts block (+1 in negated →
   −1 in output), newly-absent negated facts unblock (−1 in negated →
   +1 in output).

6. **`evaluateRuleSemiNaive` does not handle negation as a delta source.**
   It already handles positive atoms as delta sources via `deltaIdx`. The
   same function can be generalized: the body element at `deltaIdx` knows
   its own kind (`atom` vs `negation`), so the switch branch can dispatch
   to the appropriate handler without a separate function or parameter.

7. **No `getNegationAtomIndices` helper.** Needed to enumerate negation
   body elements for differential processing per rule.

8. **Routing logic in `evaluateStratumFromDelta`** dispatches to
   `wipeAndRecompute` based on boolean flags. Must be replaced with the
   unified loop.

9. **`step()` threads `retractionsPresent` flag** through strata. Must be
   removed — each stratum handles its own retractions via the unified loop.

## Architecture

### Dual-Weight `Relation`

Each `Relation` entry stores two weights:

- **`weight`** — the true Z-set multiplicity. This is the mathematically
  fundamental quantity: the number of independent derivation paths minus
  the number of retraction paths. `addWeighted` sums into this field.
  It may be > 1 (multiple derivation paths), 0 (all paths retracted),
  or transiently < 0 during an iteration (retraction delta overshoot,
  clamped by `applyDistinct`).

- **`clampedWeight`** — the post-`distinct` presence signal: `weight > 0
  ? 1 : 0`. Updated **eagerly** by `addWeighted` (for mid-iteration
  visibility) and **authoritatively** by `applyDistinct` (for the
  negative-floor clamp). This is the derived quantity that consumers see.
  `has()`, `tuples()`, `size`, `weightedTuples()`, `isEmpty()`, and
  `evaluateNegation` all read `clampedWeight`.

The naming follows the Z-set convention: `weight` is the natural name
for multiplicity, matching `Substitution.weight` (provenance product)
and the DBSP literature. `clampedWeight` signals that it's a derived
quantity — the output of `distinct(weight)`.

**Invariants:**
- After `applyDistinct`: `weight >= 0` (negatives clamped to 0 as floor).
  `clampedWeight = weight > 0 ? 1 : 0` for all dirty entries.
- Between outer time steps (between `step()` calls): all entries satisfy
  `weight >= 0` and `clampedWeight ∈ {0, 1}`.
- During an iteration (before `applyDistinct`): `weight` may be any
  integer. `clampedWeight` is **eagerly kept in sync** by `addWeighted`:
  `clampedWeight = newWeight > 0 ? 1 : 0` after every mutation. This
  ensures mid-iteration visibility — facts added by one rule are
  immediately visible to subsequent rules via `weightedTuples()`.

**`addWeighted` behavior change:**
- Sums into `weight` (as before).
- Eagerly sets `clampedWeight = newWeight > 0 ? 1 : 0`.
- Prunes entries at `weight == 0` (as before).

**`applyDistinct` behavior change (simplified):**
- Only handles the **negative-floor**: if `weight < 0`, set `weight = 0`
  (which prunes the entry via `addWeighted`). The `clampedWeight` is
  already 0 (set eagerly by the `addWeighted` call that made `weight`
  negative).
- `weight > 1` entries: **no action**. `weight` is preserved (true
  multiplicity). `clampedWeight` is already 1 (set eagerly).
- This is a significant simplification — `applyDistinct` degenerates to
  a negative-floor pass over dirty entries.

Note: DBSP's `distinct` operator is `distinct(w)(x) = max(0, w(x))` —
it floors negatives to 0 but does NOT clamp positives to 1. This is
NOT the same as 0/1 clamping. The 0/1 clamping from Plan 006.1 was a
set-oriented approximation that worked only because weights were always
0 or 1 after clamping. With true Z-set multiplicities, we must use the
DBSP definition: negative-floor only. See "Semi-Naive Self-Join
Double-Counting" in §Learnings for why this is safe.

**Key consequence:** `weightedTuples()` filters on `clampedWeight > 0`
and returns `clampedWeight` as the weight value (always 1). Positive atom
joins via `evaluatePositiveAtom` against the accumulated DB see weight 1
for all present entries, preventing weight explosion in recursive rules.
`allWeightedTuples()` returns the true `weight` — used for delta DBs.

**Pruning:** `addWeighted` continues to prune entries at `weight == 0`.
For new entries, `clampedWeight` is set eagerly by `addWeighted` (not
deferred to `applyDistinct`). Delta databases (transient, never run
through `applyDistinct`) have `clampedWeight` set eagerly by `addWeighted`
like any other database — though delta DBs are typically read via
`allWeightedTuples()` which returns the true `weight`, not `clampedWeight`.

### Deferred-Delta Evaluation (DBSP Asymmetric Join)

**Critical correctness requirement.** Standard semi-naive evaluation
iterates over delta positions for rules with multiple body atoms on the
same predicate (self-joins). This produces double-counted weights when
the delta overlaps with `fullDb`. For example, `superseded(A,S) :-
active_value(A,...), active_value(B,...), ...` with delta containing
both alice and bob produces `superseded(alice)` through BOTH deltaIdx=0
and deltaIdx=1, giving weight 2 for a fact with only 1 genuine
derivation. On retraction, only 1 of the 2 paths reverses, leaving
weight 1 — the fact incorrectly survives.

The fix comes from DBSP's incremental join formulation. For a binary
join `A ⋈ B` (where A and B may be the same relation P), the correct
incremental update is:

```
Δ(A ⋈ B) = ΔA ⋈ B_new + A_old ⋈ ΔB
```

Where `A_old = A_new - ΔA` is the **pre-update** state. The asymmetry
ensures each (a, b) pair is counted exactly once. Our current code uses
`A_new ⋈ ΔB + ΔA ⋈ B_new`, which double-counts pairs where both
a ∈ ΔA and b ∈ ΔB.

**Implementation: deferred delta application.** The key insight is that
we don't need to snapshot or clone the DB. Instead:

1. `step()` does NOT apply the input delta to `db` before calling
   `evaluateStratumFromDelta`. It passes `db` (= `P_old`) and
   `inputDelta` (= ΔP) separately.

2. `evaluateStratumFromDelta` constructs `P_new` for predicates in the
   delta by merging: for each predicate in the delta, create a temporary
   `Database` containing `db[pred] + delta[pred]`. This is O(|delta|).

3. `evaluateRuleDelta` gains a `fullDbNew` parameter (in addition to
   `fullDb` which now means `P_old`). For position `j` in the rule body:
   - `j == deltaIdx`: use `delta` (with `allEntries: true`)
   - `j < deltaIdx` AND same predicate as delta: use `fullDbNew`
   - `j > deltaIdx` OR different predicate: use `fullDb` (= `P_old`)

4. After the seed phase, apply the input delta to `db` so that
   subsequent iterations see the current state.

5. For the **iteration** phase (intra-stratum derived facts), the same
   asymmetric rule applies. `applyDerivedFact` applies facts to `db`
   and adds them to `nextDelta`. At the start of each iteration, `db`
   is already `P_new` (facts were applied). The `currentDelta` contains
   only the newly-derived facts from the previous iteration. Since
   `currentDelta ⊆ db` (applied), we construct `P_old` for iteration
   by noting `P_old = db - currentDelta` for the affected predicates.
   In practice, for the iteration phase we construct a `dbOld` view
   by subtracting `currentDelta` entries from `db` for self-join
   predicates.

This approach is O(|Δ|) for constructing the merged/subtracted views,
not O(|DB|) for cloning. The asymmetric join is computed per-rule
per-delta-position with no additional passes.

### Unified Semi-Naive Loop

Every stratum — positive-only, negation, or mixed — uses the same loop:

```
evaluateStratumFromDelta(rules, db_old, inputDelta):
  dirty = new DirtyMap

  // Construct db_new for predicates in delta (O(|delta|)).
  db_new = merge(db_old, inputDelta)

  // Seed: for each rule, evaluate delta-driven from inputDelta.
  // Uses asymmetric join: positions before deltaIdx use db_new,
  // positions after deltaIdx use db_old.
  currentDelta = seed(rules, db_old, db_new, inputDelta, dirty)

  // Apply inputDelta to db (db_old becomes db_current).
  applyDelta(db, inputDelta)
  applyDistinct(db, dirty)

  // Iterate until convergence.
  while currentDelta has any entries:
    // For self-join predicates, construct db_old = db - currentDelta.
    db_old_iter = subtract(db, currentDelta)  // O(|currentDelta|)

    nextDelta = empty
    for rule in rules:
      // Positive atom delta sources (asymmetric semi-naive)
      for deltaIdx in positiveAtomIndices(rule):
        derived = evaluateRuleDelta(rule, db_old_iter, db,
                                    currentDelta, deltaIdx)
        for wf in derived: applyDerivedFact(wf, db, dirty, nextDelta)

      // Negation atom delta sources (differential negation)
      for negIdx in negationAtomIndices(rule):
        if currentDelta has entries for negated predicate at negIdx:
          derived = evaluateRuleDelta(rule, db_old_iter, db,
                                      currentDelta, negIdx)
          for wf in derived: applyDerivedFact(wf, db, dirty, nextDelta)

    applyDistinct(db, dirty)
    currentDelta = nextDelta

  return extractDelta(db, dirty)
```

Note: both the positive and negation delta paths call the **same**
`evaluateRuleDelta` function. The element at `deltaIdx` knows its own
kind and dispatches accordingly (see below).

The asymmetric join is the key correctness mechanism. Without it,
self-joins double-count and `applyDistinct`'s negative-floor-only
behavior would leave inflated weights. With it, each derivation path
is counted exactly once, making negative-floor `distinct` correct.

### Unified `evaluateRuleDelta` — Asymmetric Join

`evaluateRuleSemiNaive` is renamed to `evaluateRuleDelta` and extended
with two changes: (1) a new `case 'negation'` branch for differential
negation, and (2) asymmetric join support via a `fullDbNew` parameter.

The body element at `deltaIdx` already carries its kind (`atom` vs
`negation`), so the function needs no `deltaKind` parameter — it reads
`element.kind` and dispatches:

- `case 'atom'` at `deltaIdx`: evaluate positive atom against `delta`
  (with `allEntries: true` to see negative-weight entries).
- `case 'negation'` at `deltaIdx`: evaluate differential negation against
  `delta` via `evaluateDifferentialNegation`.
- `case 'atom'` at non-delta positions:
  - If `j < deltaIdx` AND the atom references a predicate present in the
    delta: use `fullDbNew` (post-update state).
  - Otherwise: use `fullDb` (pre-update state, = `P_old`).
- Non-delta negations: boolean `evaluateNegation` against `fullDbNew`
  (the current state of the negated relation matters).
- Guards and aggregations: unchanged.

The signature becomes:

```
evaluateRuleDelta(rule, fullDbOld, fullDbNew, delta, deltaIdx):
    WeightedFact[]
```

Where `fullDbOld` is the pre-update state (`P_old`) and `fullDbNew` is
the post-update state (`P_new = P_old + delta`). For predicates not
present in the delta, `fullDbOld` and `fullDbNew` are identical (the
same underlying `Database` object).

The asymmetric dispatch for non-delta positive atoms is:

```
case 'atom': {
  if (isDeltaSource) {
    subs = evaluatePositiveAtom(element.atom, delta, subs, true);
  } else {
    // Asymmetric join: positions before deltaIdx use P_new,
    // positions after use P_old. This prevents double-counting.
    const db = (i < deltaIdx && deltaHasPredicate(element.atom.predicate))
      ? fullDbNew : fullDbOld;
    subs = evaluatePositiveAtom(element.atom, db, subs);
  }
  break;
}
```

`evaluateRuleSemiNaive` is retained as a deprecated alias that passes
`fullDb` as both `fullDbOld` and `fullDbNew` (backward compatible but
without asymmetric join correctness — only safe for rules without
self-joins on delta predicates).

### Differential Negation

`evaluateDifferentialNegation(atom, delta, subs)` is the leaf function
for the negation branch. It matches the negation atom against the
**delta** (not the full DB) using `allWeightedTuples()`. For each
matching entry in the delta:

- If delta weight is +1 (fact newly appeared in negated relation): this
  binding is now blocked → emit substitution with `weight *= -1`.
- If delta weight is −1 (fact disappeared from negated relation): this
  binding is now unblocked → emit substitution with `weight *= +1`.

In both cases the formula is: `output_weight = sub.weight × (-deltaWeight)`.

The sign inversion encodes the semantics of negation-as-failure:
appearance of a negated fact *removes* derivations; disappearance of a
negated fact *adds* derivations.

### Bidirectional `applyDerivedFact`

The current function detects absent→present crossings and seeds
`nextDelta` with +1. The fix adds present→absent detection: when a
fact's `clampedWeight` crosses from 1 to 0, seed `nextDelta` with −1
via `addWeightedFact`. This allows cascading retractions through
recursive joins (e.g., transitive closure losing an edge).

The zero-crossing check compares pre- and post-`touchFact` values of
`weight > 0` (which is equivalent to `clampedWeight` changing, since
`clampedWeight = weight > 0 ? 1 : 0`). A fact that drops from
`weight = 2` to `weight = 1` does NOT cross zero — no retraction
propagated. Only when `weight` reaches 0 does `clampedWeight` change
and a retraction enter the delta.

### `evaluatePositiveAtom` — Conditional `allEntries` Parameter

`evaluatePositiveAtom` must use different entry-fetching strategies
depending on whether it's matching against the accumulated DB or a
delta DB:

- **Accumulated DB (non-delta positions):** Use `weightedTuples()`, which
  returns entries with `clampedWeight > 0` and reports `clampedWeight`
  (always 1) as the weight. This prevents weight explosion in recursive
  rules — joins see weight 1, not the true multiplicity.

- **Delta DB (at `deltaIdx`):** Use `allWeightedTuples()`, which returns
  all entries including negative-weight retractions. The true `weight`
  (including −1) flows through the provenance product, producing
  negative-weight substitutions that represent retracted derivations.

`evaluatePositiveAtom` gains a default-false `allEntries?: boolean`
parameter. When `false` (default): uses `weightedTuples()` — identical
to current behavior. When `true`: uses `allWeightedTuples()`. This is
NOT a public API concern — it's used only by `evaluateRuleDelta` at the
delta index.

This is the minimal change. Non-delta positions continue to use
`weightedTuples()` (weight 1, no explosion). Delta positions use
`allWeightedTuples()` (true weight, including negatives).

**Why not unconditional `allWeightedTuples()`:** For the accumulated DB
after `applyDistinct`, entries may have `weight > 1` (true multiplicity
preserved by the dual-weight scheme). Joining against weight 2 would
produce substitutions with weight 2, compounding through recursive rules.
DBSP's recursive circuit applies `R` to the output of `distinct`, meaning
joins must see the clamped weight (0/1), not the raw multiplicity.

### Size Checks — `Database.hasAnyEntries()`

Multiple `Database.size` checks must be replaced with a check that
accounts for negative-weight entries:

1. **Convergence check**: `currentDelta.size > 0` in the semi-naive loop.
   A delta containing only −1 entries has `size === 0` and would terminate
   the loop prematurely.

2. **Seed guard**: `inputDelta.size > 0` in the seed phase. If `inputDelta`
   contains only −1 entries, `size === 0` and the seed phase is skipped
   entirely — retractions in the input would silently do nothing. This is
   **correctness-critical**.

3. **Early exit check**: `inputDelta.size === 0 && currentDelta.size === 0`.
   Same issue.

All three are replaced with `Database.hasAnyEntries()`, a new method that
returns true if any relation has any stored entries regardless of weight
sign. This is an O(1) early-exit scan (find first non-empty relation).

### Aggregation Strata — Scoped Limitation

Aggregation (`min`, `max`, `count`, `sum`) is a group-by boundary that
resets provenance to weight 1. Differential aggregation would require
maintaining per-group state to detect when a group's aggregate value
changes — essentially a mini-evaluator per group. This is substantially
more complex than differential negation and is not needed for the default
rules (no aggregation in default LWW or Fugue rules).

This plan **retains wipe-and-recompute for aggregation-only strata** as
a scoped, documented limitation. The routing for this is minimal: check
if a stratum has aggregation body elements. If yes, wipe-and-recompute.
If the stratum has negation but no aggregation, use the unified loop.
This is a much narrower condition than the current "any negation or
aggregation" check.

The wipe-and-recompute code is retained but renamed to
`recomputeAggregationStratum` and scoped specifically to aggregation.
The generic `wipeAndRecompute` function is deleted.

### Rule Changes — Retain `diffDerivedSnapshots`

Rule changes are a fundamentally different operation from fact changes.
When rules change, the entire set of derivable facts may differ — old
derived facts may be underivable under the new rules, and new facts may
be derivable that weren't before. The unified semi-naive loop is designed
for incremental *fact* changes against stable rules, not for reasoning
about changes to the rules themselves.

The rule-change path in `step()` retains its current architecture:
wipe all derived facts, replay all strata from scratch, diff snapshots
via `diffDerivedSnapshots`. Rule changes are rare (custom constraint
addition/retraction) and inherently O(|DB|). The snapshot-and-diff cost
is negligible compared to the full replay.

`diffDerivedSnapshots` is **retained** for the rule-change path.

### Convergence Argument

In stratified Datalog with dual-weight `distinct`:

- **Lower-stratum predicates are converged** before the current stratum
  begins. The differential negation pass fires only for entries in the
  input delta whose predicates match negation atoms — these are from
  lower strata and don't change during iteration.
- **Intra-stratum iteration** involves only the stratum's own derived
  predicates, which don't appear in its own negation atoms (cyclic
  negation is rejected by stratification).
- **Positive sub-loop convergence**: Each +1 adds a new fact or increases
  an existing fact's `weight` (bounded by finite derivation paths).
  Each −1 decreases `weight`. `applyDistinct` clamps negative `weight`
  to 0 (floor) but preserves `weight > 1` (multiplicities). A retraction
  propagates only when `weight` crosses zero (i.e., `clampedWeight`
  changes from 1 to 0). The set of facts with `clampedWeight > 0`
  decreases monotonically per retraction iteration. No infinite
  oscillation is possible.
- **Multi-path derivations**: A fact with `weight = 2` (two derivation
  paths) drops to `weight = 1` on one retraction. `clampedWeight` stays
  1. No retraction propagated — the fact survives. Only when `weight`
  reaches 0 does `clampedWeight` change to 0, triggering retraction
  propagation. This is correct: the fact has lost all support.
- **Recursive rules with retractions** (e.g., transitive closure losing
  an edge): Each iteration can only retract facts whose `weight` crosses
  zero. The total set of zero-crossing-eligible facts is bounded by the
  current set of facts with `weight > 0`. Convergence follows from the
  finite domain.

### Functional Core / Imperative Shell

The split is preserved from Plan 006.1:

- **Functional core** (`evaluateStratumFromDelta`): Given rules, a mutable
  db, and an input delta, run the unified semi-naive loop to convergence
  and return the output delta. Independently testable.
- **Imperative shell** (`createEvaluator` + `step`): Manages accumulated
  state, affected-strata computation, inter-stratum delta propagation.
  Simplified by removing `retractionsPresent` flag threading.

Within each semi-naive iteration, the computation is cleanly separated:

1. **Compute phase**: `evaluateRuleDelta` (pure — takes immutable inputs,
   returns `WeightedFact[]`).
2. **Apply phase**: `applyDerivedFact` + `touchFact` (mutates db `weight`
   and dirty map, produces nextDelta based on `weight` zero-crossings).
3. **Clamp phase**: `applyDistinct` (clamps negative `weight` to 0;
   `clampedWeight` is already in sync from eager updates in step 2).

## Transitive Effect Analysis

### Dual-weight `Relation` entry change

The internal entry type gains `clampedWeight`. This affects:

**`Relation` methods (Phase 0):** All presence-checking methods (`has`,
`tuples`, `size`, `weightedTuples`, `isEmpty`) switch from `weight > 0`
to `clampedWeight > 0`. `allWeightedTuples()` continues to return the
true `weight`. `getWeight()` returns the true `weight`. `clone()` copies
both fields. `union()` and `difference()` use `clampedWeight` for
presence.

**`applyDistinct` (Phase 0):** No longer mutates `weight` for > 1 values.
Sets `clampedWeight` and only clamps negative `weight` to 0 (floor).
After `applyDistinct`, `weight >= 0` and `clampedWeight ∈ {0, 1}` for
all dirty entries.

**`touchFact` (Phase 0):** Records `preWeight` as the true `weight`
before mutation. This is correct — `extractDelta` checks whether `weight`
crossed zero (which is equivalent to `clampedWeight` changing, since
`addWeighted` eagerly keeps `clampedWeight` in sync).

**`extractDelta` (no change):** Compares `preWeight > 0` to
`currentWeight > 0`. Since `getWeight()` returns the true `weight`, and
zero-crossing of `weight` ≡ change in `clampedWeight`, the logic is
unchanged and correct.

**Backward compatibility:** After `applyDistinct`, `clampedWeight` for
present entries is 1, and `weightedTuples()` returns `{ tuple, weight: 1 }`.
In Plan 006.1, after `applyDistinct`, `weight` was 1 for present entries
(clamped from any higher value). So `weightedTuples()` returned
`{ tuple, weight: 1 }` identically. No behavioral change for any code
that reads through the `Relation` API.

**`evaluatePositiveAtom` (Phase 1):** Gains an `allEntries` parameter.
At non-delta positions, uses `weightedTuples()` (returns `clampedWeight`
= 1) — identical to current behavior, prevents weight explosion in
recursive rules. At the delta position in `evaluateRuleDelta`, uses
`allWeightedTuples()` (returns true `weight`, including negatives) — the
provenance product `sub.weight × tuple.weight` correctly produces
negative-weight substitutions for retracted delta entries.

### `applyDerivedFact` bidirectional propagation

Previously, only absent→present crossings seeded `nextDelta`. Now
present→absent crossings also seed `nextDelta` with −1.

**Risk:** Over-retraction when a fact has multiple derivation paths.
Mitigation: The dual-weight scheme ensures `weight` tracks true
multiplicity. A retraction of one path decrements `weight` (e.g., 2 → 1)
without crossing zero. `applyDerivedFact` checks `weight > 0` before and
after (via `prevWeight` and `newWeight` from `touchFact`) — no
zero-crossing, no retraction propagated. `addWeighted` eagerly sets
`clampedWeight = 1` (since `weight = 1 > 0`), so the fact remains visible
to subsequent rules. Only when `weight` reaches 0 does `clampedWeight`
become 0 and the retraction cascade.

**Risk:** Infinite oscillation between +1 and −1. This cannot happen
because: (a) `applyDistinct` clamps negative `weight` to 0 (floor), so
each iteration starts from a non-negative state; (b) the delta for each
iteration is strictly derived from the previous delta via rule evaluation,
and the set of derivable facts is finite; (c) `weight` is a non-negative
integer after `applyDistinct`, and retractions can only decrement it,
bounding the retraction cascade; (d) `clampedWeight` is eagerly kept in
sync by `addWeighted`, so visibility is always consistent with `weight`.

### Size checks → `hasAnyEntries()`

Three `Database.size`-based checks change behavior:

1. **Convergence**: `currentDelta.size > 0` → `currentDelta.hasAnyEntries()`.
   Previously, a delta with only −1 entries terminated the loop. Now it
   continues, allowing retraction cascades.

2. **Seed guard**: `inputDelta.size > 0` → `inputDelta.hasAnyEntries()`.
   Previously, a retraction-only input delta skipped the seed phase
   entirely. Now it seeds properly, allowing retraction propagation.
   **Correctness-critical fix.**

3. **Early exit**: `inputDelta.size === 0 && currentDelta.size === 0` →
   `!inputDelta.hasAnyEntries() && !currentDelta.hasAnyEntries()`.

### `evaluateStratumFromDelta` signature change

The `hasNegOrAgg` and `inputHasRetractions` parameters are removed.

**Call sites in `evaluator.ts`:** All calls in `step()` must be updated.
There are approximately 4 call sites: the main incremental loop, the
first-step-with-no-facts path, the rule-change replay path, and the
batch wrapper path.

**Test call sites:** ~7 calls in `tests/datalog/evaluator.test.ts` pass
`hasNegOrAgg` as a positional boolean. All must be updated.

### `evaluateRuleSemiNaive` → `evaluateRuleDelta` rename

All callers of `evaluateRuleSemiNaive` must be updated. There are ~6 call
sites in `evaluator.ts` (the seed phase and the iteration loop). If a
re-export alias is used, existing external callers (e.g., test utilities)
continue to work.

### Removal of `retractionsPresent` flag in `step()`

The flag currently forces higher strata to wipe-and-recompute when lower
strata produce −1 deltas. Removing it means higher strata process the
−1 deltas via the unified loop. This is the intended behavior — the
unified loop handles retractions.

**Risk:** If the unified loop is incorrect, stale derived facts could
remain. Mitigation: the three-way oracle test catches any divergence
between batch (fresh evaluator) and incremental (accumulated state).
The three-value LWW test specifically catches the over-retraction bug.

### Aggregation strata retained as wipe-and-recompute

Strata with aggregation body elements continue to wipe-and-recompute.
The check narrows from `hasNegOrAgg` to `hasAggregation`. Strata with
negation but no aggregation now use the unified loop.

**Default rules:** No aggregation in LWW or Fugue rules. Unaffected.

**User-defined rules with aggregation:** Continue to work correctly via
wipe-and-recompute. Future plan can address differential aggregation.

### `Database.hasAnyEntries()` addition

New method on `Database`. No existing method changes. Used in the
semi-naive convergence check, seed guard, and early exit check. No
transitive effects beyond those checks.

### Resolution extraction

`winnerFactsToResolution` and `fuguePairFactsToResolution` read from the
output delta `Database` using `allWeightedTuples()`. These are unchanged
— the output delta is produced by `extractDelta`, which emits +1 and −1
entries. The resolution extraction logic is already correct for both signs.

### Rule-change path

`diffDerivedSnapshots` is **retained**. The rule-change path continues
to wipe all derived facts, replay strata, and diff snapshots. The only
change is that the per-stratum `evaluateStratumFromDelta` call within
the replay loop drops the `hasNegOrAgg` argument (the stratum function
determines aggregation internally).

### `evaluate.ts` exports and signature changes

New exports: `getNegationAtomIndices`, `evaluateDifferentialNegation`,
`evaluateRuleDelta`. `evaluateRuleSemiNaive` remains as a re-export alias
or is removed after updating all call sites. Added to `datalog/index.ts`.

`evaluatePositiveAtom` gains an optional `allEntries?: boolean` parameter
(default false). This is backward-compatible — all existing callers pass
no fourth argument and get the existing `weightedTuples()` behavior.

## Learnings

### `distinct` Destroys Multiplicity — The Three-Value LWW Bug

The correctness review uncovered that the Plan 006.1 approach of clamping
the single `weight` field to 0/1 via `applyDistinct` is **incorrect for
incremental retraction** when facts have multiple derivation paths.

Example: LWW with three values. `superseded(alice, S)` is derived by
both `supersededByLamport` (bob supersedes alice) and `supersededByLamport`
(charlie supersedes alice). The `weight` is 2 pre-distinct, clamped to 1.
Retracting charlie produces a −1 delta, dropping the weight from 1 to 0.
But alice is still superseded by bob — the retraction is incorrect.

The DBSP §5 nested-stream construction requires maintaining the
**pre-distinct** accumulated weights (∫R) separately from the
**post-distinct** presence signal (distinct(∫R)). The incremental distinct
operator is `distinct^Δ = distinct(∫R_new) − distinct(∫R_old)`, which
correctly handles multiplicity changes that don't cross zero.

The fix is the dual-weight scheme in Phase 0: `weight` stores the true
Z-set multiplicity, `clampedWeight` stores the presence signal.
`applyDistinct` sets `clampedWeight` from `weight` without clamping
`weight` (except for the negative-to-zero floor). This lets the evaluator
distinguish "one of two paths retracted" (weight 2 → 1, no output delta)
from "all paths retracted" (weight 1 → 0, emit −1 delta).

This bug would have been caught by the three-value LWW test and the
alternative-support reachability test proposed in Phase 4 — but only
after implementation, not during planning. The mathematical correctness
review caught it before any code was written.

### Semi-Naive Self-Join Double-Counting and the DBSP Asymmetric Join

**Discovered during implementation.** A first implementation attempt
applied the dual-weight scheme (negative-floor-only `applyDistinct`)
but used the standard semi-naive delta-position iteration. This passed
all existing tests for insertion but failed for the two-value LWW
retraction case:

- Insert alice (L=10) and bob (L=20). Both appear in the input delta.
- `supersededByLamport` has two `active_value` body atoms (self-join).
- deltaIdx=0: alice from delta, bob from fullDb → `superseded(alice): +1`
- deltaIdx=1: alice from fullDb, bob from delta → `superseded(alice): +1`
- Total weight: 2. But the genuine derivation count is 1 (only one
  (alice, bob) pair exists).

This is semi-naive double-counting: the same logical derivation fires
through both delta positions because the delta overlaps with `fullDb`
(since `step()` applies the delta to `db` before calling the stratum
evaluator).

On retraction (retract bob), only one delta position produces −1,
bringing the weight from 2 to 1. No zero-crossing, no retraction
propagated. The `winner` rule never sees the retraction.

**Root cause:** Standard semi-naive computes
`ΔA ⋈ B_new + A_new ⋈ ΔB` where A = B = P (the same relation). When
ΔP ⊆ P_new (true because delta was applied), pairs where both elements
are in ΔP get counted twice. DBSP's correct formulation is
`ΔA ⋈ B_new + A_old ⋈ ΔB` — the asymmetric join, where `A_old = A_new
− ΔA` is the pre-update state.

**A naive fix** of restoring the old 0/1 clamping in `applyDistinct`
makes the two-value case work but breaks the three-value case (where
weight 2 from genuinely independent paths is correctly needed).

**The correct fix:** Deferred-delta evaluation. Don't apply the delta
to `db` before the seed phase. Pass `db` (= `P_old`) and `inputDelta`
separately. Construct `P_new` by merging for the affected predicates
(O(|delta|)). In `evaluateRuleDelta`, use `P_new` for positions before
`deltaIdx` and `P_old` for positions after. This counts each derivation
exactly once:

- deltaIdx=0: CnId=alice from delta, CnId2 from `P_new` (bob). +1. ✓
- deltaIdx=1: CnId from `P_old` (empty — alice/bob not yet applied),
  CnId2=bob from delta. No match. 0.
- Total weight: 1. Correct.

On retraction with `P_old = {alice, bob}`:
- deltaIdx=1: CnId=alice from `P_old`, CnId2=bob from delta (−1). −1. ✓
- Total: −1. Weight goes from 1 to 0. Zero-crossing. Retraction
  propagates correctly.

**Key insight:** `P_old = P_new − ΔP`. We don't need O(|DB|) snapshots.
For the seed phase, just don't apply the delta before evaluation. For the
iteration phase, construct `P_old = db − currentDelta` for self-join
predicates, which is O(|currentDelta|).

**Lesson:** DBSP's `distinct` operator is `max(0, w)` (negative floor),
NOT `clamp(0, 1, w)`. The 0/1 clamping is a set-oriented approximation
that masks semi-naive double-counting. To use true Z-set multiplicities,
the semi-naive evaluation itself must be correct — which requires the
asymmetric join formulation.

### `weight` Is the Mathematically Natural Name

In DBSP, "weight" means Z-set multiplicity — the count of independent
derivation paths. `Substitution.weight` already uses this convention
(the provenance product through joins). Naming the clamped 0/1 signal
`weight` and the true multiplicity something else (like `rawWeight`)
would be semantically backwards. The rename to `clampedWeight` for the
derived quantity aligns with the DBSP literature and with the existing
`Substitution.weight` semantics.

### Joins Must Use Post-Distinct Weights to Prevent Explosion

In recursive rules (e.g., transitive closure), joins against the
accumulated DB must use post-distinct weights (0/1) to prevent weight
explosion. If `reachable(a,c)` has `weight = 2` and we join against it
with the true weight, derivative facts get multiplied by 2, which
compounds per iteration. DBSP's recursive circuit applies `R` to the
output of `distinct`, not to the raw accumulated state. In our
implementation, `evaluatePositiveAtom` with `allEntries: false` (default)
uses `weightedTuples()` which returns `clampedWeight` (always 1) — used
for non-delta joins. With `allEntries: true`, it uses `allWeightedTuples()`
which returns the true `weight` — used only at the delta index in
`evaluateRuleDelta`. This is why `evaluatePositiveAtom` cannot switch to
`allWeightedTuples()` unconditionally.

## Alternatives Considered

### Single Weight with `applyDistinct` Clamping to 0/1 (Plan 006.1 Status Quo)

Keep the single `weight` field and clamp to 0/1 after each iteration.
Rejected because: this destroys multiplicity information required for
correct retraction. The three-value LWW bug demonstrates that a fact
with two derivation paths is incorrectly retracted when one path is
removed. This is not an edge case — it's the common case with 3+
collaborators in the same slot.

### Separate `evaluateRuleDifferentialNegation` Function

Create a parallel function to `evaluateRuleSemiNaive` that handles
negation as the delta source. Rejected because: the two functions share
~90% of their code (same loop, same switch, same body-element dispatch
for all non-delta elements). The body element at `deltaIdx` already knows
its own kind (`atom` vs `negation`), so the switch branch can dispatch
to the correct handler without a parallel function or `deltaKind`
parameter. One function, one loop, one extra ternary in `case 'negation'`.

### Maintain `wipeAndRecompute` for Negation, Fix Only Positive Strata

Fix `applyDerivedFact` and `evaluatePositiveAtom` to handle retractions
in positive strata, but keep wipe-and-recompute for negation strata.
Rejected because: (a) the differential negation implementation is
straightforward once positive retraction propagation works, (b) negation
strata are exactly where the performance benefit matters most (the LWW
winner rule is in a negation stratum), (c) keeping two paths perpetuates
the conditional routing complexity.

### Eliminate `diffDerivedSnapshots` for Rule Changes Too

Use the unified semi-naive loop for rule changes by feeding a "full"
input delta. Rejected because: rule changes alter the set of derivable
facts in ways that can't be incrementally discovered — old derived facts
may become underivable under new rules, and the semi-naive loop has no
mechanism to discover "this fact can no longer be derived." Wipe-then-
replay is the correct architecture for rule changes. Rule changes are
rare and inherently O(|DB|); the snapshot-and-diff cost is negligible.

### Full Differential Aggregation

Implement incremental aggregation by tracking per-group state and
detecting when aggregate values change. Rejected for this plan because:
(a) no default rules use aggregation, (b) differential aggregation
requires maintaining group membership sets, which is substantially more
complex, (c) the aggregation wipe-and-recompute is bounded by the size
of the aggregation stratum, which is typically small, (d) it can be
addressed in a future plan when Layer 3+ queries create large aggregation
strata.

### Per-Fact Provenance DAGs Instead of Z-Set Weights

Track which specific ground facts support each derivation, enabling
precise retraction by walking the DAG. Rejected because: (a) DBSP's
Z-set arithmetic subsumes this via weight counting, (b) per-fact DAGs
are O(|derivation paths|) in memory, which can be exponential for
recursive rules, (c) the dual-weight scheme keeps weights at O(1) per
fact, (d) the weight infrastructure is already built.

### Treat Negation Body Elements as Anti-Joins with Z-Set Semantics

Model `not P(X)` as a Z-set anti-join: `subs ⋈ (1 - distinct(P))`.
This is mathematically clean but requires materializing `1 - distinct(P)`
for the negated relation, which is the complement set — potentially
infinite or at least very large. Rejected in favor of the targeted
differential approach which only processes the *changed* entries in the
negated relation.

### Defer to Plan 008 (Query Layer)

Address differential negation when user queries create large negation
strata. Rejected because: (a) the fix is localized to two files and
~200 lines of new code, (b) it eliminates ~200 lines of conditional
routing and wipe-and-recompute code, (c) it makes the evaluator
conceptually simpler (one algorithm, not three paths), (d) Plan 008
would inherit the complexity if deferred.
