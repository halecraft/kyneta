# Schema Interpreter Algebra

> A schema is a functor. Each use case is a natural transformation.

## 1. The Observation

The `@loro-extended/change` package builds proxy trees over Loro CRDTs.
But the same tree shape is walked at least **ten** different ways:

| Interpretation | text → | counter → | list\<S\> → | struct\<K,V\> → |
|---|---|---|---|---|
| **Plain** | `string` | `number` | `T[]` | `{ k: V }` |
| **Writable** | `TextRef` | `CounterRef` | `ListRef<S>` | `StructRef<S>` |
| **Feed** | `Feed<string, TextAction>` | `Feed<number, ReplaceAction>` | `Feed<self, ListAction>` | `Feed<self, MapAction>` |
| **Zero** | `""` | `0` | `[]` | `{ k: zero(V) }` |
| **Path** | `PathSelector<string>` | `PathSelector<number>` | `ListPathNode<S>` | `StructPathNode<S>` |
| **Overlay** | `merge(crdt, zero)` | `merge(crdt, zero)` | `items.map(merge)` | `{ k: merge(crdt[k], zero[k]) }` |
| **Conversion** | `new LoroText(s)` | `new LoroCounter(n)` | `new LoroList(...)` | `new LoroMap(...)` |
| **Reconstruct** | `flat[name]` | `flat[name]` | `flat[name]` | `{ k: reconstruct(k) }` |
| **Assign** | `ref.update(s)` | `ref.increment(Δ)` | `ref.push(...)` | `ref[k].set(v)` |
| **Serialize** | `ref.toString()` | `ref.get()` | `ref.map(toJSON)` | `{ k: ref[k].toJSON() }` |

Each row is implemented as a separate `switch (shape._type)` dispatch
site: `derivePlaceholder`, `overlayPlaceholder/mergeValue`, `createPathNode`,
`convertInputToRef`, `reconstructFromFlattened`, `createContainerTypedRef`,
`assignPlainValueToTypedRef`, `serializeRefToJSON`/per-class `toJSON()`,
and the lazy property creation in `DocRefInternals`/`MapBasedRefInternals`.

They all walk the same shape. They differ only in what they *do* at each
node. The goal of this package is to extract the shared structure.

## 2. Schema as Syntax

A schema is a recursive tree grammar. We define it as an initial algebra —
the free structure generated by these node constructors:

```
SchemaF<A> =
  | Text
  | Counter
  | Value<T>             -- plain scalars (string, number, boolean, etc.)
  | List<A>              -- ordered collection of A
  | MovableList<A>       -- ordered collection with move semantics
  | Record<A>            -- string-keyed dynamic map of A
  | Struct<{ k₁: A, k₂: A, ... }>  -- fixed-key product
  | Tree<A>              -- hierarchical tree with A-typed node data
  | Doc<{ k₁: A, k₂: A, ... }>     -- root document (similar to Struct)
```

The **schema** is `Fix(SchemaF)` — the fixed point (recursive closure) of
this functor. A concrete schema like:

```
Schema.doc({
  title: Schema.text(),
  todos: Schema.list(Schema.string()),
})
```

is a *value* in this type: `Doc<{ title: Text, todos: List<Value<string>> }>`.

The schema carries **no runtime behavior and no defaults**. It is pure
syntax — a description of structure.

### 2.1. Value shapes as a nested algebra

`Value<T>` is not truly terminal. The `ValueShape` type is itself a
discriminated union with 13 variants (string, number, boolean, null,
undefined, uint8array, struct, record, array, union, discriminatedUnion,
any, bytes). Some of these are recursive — a struct value contains
nested value shapes, an array value contains a value shape, unions
contain multiple value shapes.

This means the catamorphism handles **two levels of recursion**: the
container-level `SchemaF` and the value-level `ValueSchemaF`. The
interpreter interface reflects this with a single `value(ctx, path, shape)`
case where the `shape` parameter carries the full `ValueShape`, and the
interpreter implementation handles the inner recursion internally.

### 2.2. Schema node kinds are closed

The schema node kinds correspond to **CRDT structural concepts**: text,
counter, list, movable list, record, struct, tree. These are the lingua
franca between backends. A Firestore backend maps the same schema nodes
to Firestore concepts (collections, documents, fields). The structural
vocabulary is stable.

**Value types**, however, are extensible. A Firestore adapter might want
`Timestamp`, `GeoPoint`, `DocumentReference`. These can be expressed as
custom value shapes without modifying the core schema node kinds, because
value shapes are terminal — interpreters handle them opaquely.

## 3. Interpreters as Algebras

An **F-algebra** is a function `F<A> → A` that collapses one layer of
structure into a result. An interpreter for our schema is an algebra
`SchemaF<A> → A` for some result type `A`:

```typescript
interface Interpreter<Ctx, A> {
  text       (ctx: Ctx, path: Path): A
  counter    (ctx: Ctx, path: Path): A
  value      (ctx: Ctx, path: Path, shape: ValueShape): A
  list       (ctx: Ctx, path: Path, item: (ctx: Ctx, index: number) => A): A
  movableList(ctx: Ctx, path: Path, item: (ctx: Ctx, index: number) => A): A
  struct     (ctx: Ctx, path: Path, fields: Record<string, () => A>): A
  record     (ctx: Ctx, path: Path, item: (ctx: Ctx, key: string) => A): A
  tree       (ctx: Ctx, path: Path, nodeData: (ctx: Ctx) => A): A
  doc        (ctx: Ctx, path: Path, fields: Record<string, () => A>): A
}
```

Note: `struct` and `doc` fields are **thunks** (`() => A`), not pre-computed
values. This preserves the laziness that the codebase depends on — today's
`DocRefInternals.propertyCache`, `MapBasedRefInternals.refCache`, and
`ListRefBaseInternals.itemCache` all demonstrate cache-on-first-access.
The interpreter decides when (and whether) to force each thunk.

The generic tree walker (`cata` — the catamorphism) applies an interpreter
to a schema, producing a result at each node:

```typescript
function interpret<Ctx, A>(
  schema: Schema,
  interp: Interpreter<Ctx, A>,
  ctx: Ctx,
  path: Path = [],
): A
```

Each of today's parallel implementations becomes a small interpreter:

### 3.1. The Plain interpreter (toJSON, value())

```
plain.text    = (doc, path) => getText(doc, path).toString()
plain.counter = (doc, path) => getCounter(doc, path).value
plain.value   = (doc, path, shape) => getMap(doc, parentPath).get(lastKey(path))
plain.list    = (doc, path, item) => range(list.length).map(i => item(doc, i))
plain.struct  = (doc, path, fields) => mapValues(fields, f => f())
plain.doc     = (doc, path, fields) => mapValues(fields, f => f())
```

### 3.2. The Zero interpreter (derivePlaceholder)

See §6 — zero is separated from the schema.

```
zero.text    = () => ""
zero.counter = () => 0
zero.value   = (_, __, shape) => typeDefault(shape)
zero.list    = () => []
zero.struct  = (_, __, fields) => mapValues(fields, f => f())
zero.doc     = (_, __, fields) => mapValues(fields, f => f())
```

### 3.3. The Writable interpreter

```
writable.text    = (ctx, path) => ({ insert, delete, update, mark })
writable.counter = (ctx, path) => ({ increment, decrement })
writable.value   = (ctx, path, shape) => PlainRef(paramsFor(ctx, path, shape))
writable.list    = (ctx, path) => ({ push, insert, delete, set })
writable.struct  = (ctx, path, fields) => createStructProxy(fields, paramsFor(ctx, path))
writable.doc     = (ctx, path, fields) => createDocProxy(fields, ctx)
```

### 3.4. The Path interpreter (PathBuilder)

```
pathBuilder.text    = (_, path) => PathSelector(segments(path))
pathBuilder.counter = (_, path) => PathSelector(segments(path))
pathBuilder.value   = (_, path) => PathSelector(segments(path))
pathBuilder.list    = (_, path, item) => ListPathNode(segments(path), item)
pathBuilder.struct  = (_, path, fields) => StructPathNode(segments(path), fields)
```

### 3.5. The Conversion interpreter (plain values → CRDT containers)

```
conversion.text    = (_, __, value) => new LoroText(value)
conversion.counter = (_, __, value) => new LoroCounter(value)
conversion.value   = (_, __, value) => value  -- passthrough
conversion.list    = (_, __, items, itemConvert) => LoroList.from(items.map(itemConvert))
conversion.struct  = (_, __, fields) => LoroMap.from(fields)
```

The pattern is clear: the tree-walking logic (recursion, lazy property
creation, caching, Proxy traps) is identical across all interpreters.
Only the leaf and branch actions differ.

## 4. Actions: The Universal Currency of Change

### 4.1. Unifying operations and deltas

Today's codebase has two separate vocabularies for change: **operations**
(developer → backend: `ref.insert(0, "hi")`) and **deltas** (backend →
observer: `{ type: "text", ops: [{ insert: "Hello" }] }`). But these
describe the same information — one is intent, the other is notification.

An **action** is the unified type. It describes a change to a schema
node. The same action structure flows in both directions:

- **Going in** (developer → backend): the action describes intent
- **Coming out** (backend → observer): the action describes what happened
- **Pure computation** (`step`): `(State, Action) → State`

### 4.2. Action types

Actions are an **open protocol** identified by a string discriminant:

```typescript
interface ActionBase {
  readonly type: string
}
```

Built-in action types use the retain/insert/delete encoding where
applicable:

```typescript
// Text: cursor-based retain/insert/delete over characters
type TextAction = { type: "text"; ops: TextActionOp[] }
type TextActionOp =
  | { retain: number }
  | { insert: string }
  | { delete: number }

// List: cursor-based retain/insert/delete over items
type ListAction<T> = { type: "list"; ops: ListActionOp<T>[] }
type ListActionOp<T> =
  | { retain: number }
  | { insert: T[] }      // carries values (not just count)
  | { delete: number }

// Map: key-level set/delete for structs, records, docs
type MapAction = { type: "map"; set?: Record<string, unknown>; delete?: string[] }

// Scalar replacement
type ReplaceAction<T> = { type: "replace"; value: T }

// Tree: structural operations
type TreeAction = { type: "tree"; ops: TreeActionOp[] }
type TreeActionOp =
  | { action: "create"; index: number }
  | { action: "delete"; index: number }
  | { action: "move"; fromIndex: number; toIndex: number }
```

Third-party backends define their own action types:

```typescript
interface IncrementAction extends ActionBase {
  readonly type: "increment"; readonly amount: number
}
interface RichTextAction extends ActionBase {
  readonly type: "richtext"; readonly ops: RichTextOp[]
}
```

### 4.3. Notifications as projection

The current delta vocabulary (`ReactiveDelta`) is a lightweight
**notification** system where `ListDelta` carries `insert: number`
(count only) and `MapDelta` carries just key names. This is efficient
for DOM patching where the consumer re-reads from the feed's head.

A notification is a **lossy projection** of an action:

```
notify : Action → Notification

notify(ListAction)    → { type: "list", ops: [...with insert as count, not values...] }
notify(MapAction)     → { type: "map", keys: [...changed key names...] }
notify(ReplaceAction) → { type: "replace" }   (value stripped)
notify(TextAction)    → TextAction             (already lightweight — ops carry content)
```

Feeds deliver full actions. Consumers that only need structural hints
can project to notifications. Consumers that want pure `step` use the
full action. This is an optimization choice, not a system constraint.

### 4.4. The natural transformation

The mapping from schema node to action type is determined by the
**backend interpreter**, not by the schema itself:

```
// Loro backend:
loroAction(Text)      = TextAction
loroAction(Counter)   = ReplaceAction<number>
loroAction(List<A>)   = ListAction<A>
loroAction(Struct<_>) = MapAction

// Hypothetical Firestore backend:
firestoreAction(Text)      = ReplaceAction<string>  -- no text CRDTs
firestoreAction(Counter)   = IncrementAction         -- native increments
firestoreAction(List<A>)   = ReplaceAction<A[]>      -- arrays replaced wholesale
firestoreAction(Struct<_>) = MapAction               -- documents are key-value
```

The schema determines the *structure*. The backend determines the *action
vocabulary*. The feed ties them together.

### 4.5. The `step` function

`step` applies an action to a plain value, producing the next plain value.
It requires no CRDT runtime — it is pure computation:

```typescript
step : (Plain<S>, Action) → Plain<S>
```

Per action type:

```
stepText("Hello", { type: "text", ops: [{ retain: 5 }, { insert: " World" }] })
→ "Hello World"

stepList([1, 2, 3], { type: "list", ops: [{ retain: 1 }, { insert: [10, 20] }, { delete: 1 }] })
→ [1, 10, 20, 3]

stepMap({ a: 1, b: 2 }, { type: "map", set: { a: 10 }, delete: ["b"] })
→ { a: 10 }

stepReplace(42, { type: "replace", value: 99 })
→ 99
```

`step` enables:

- **Optimistic UI**: apply an action locally before the backend confirms
- **Time travel**: fold a sequence of actions over an initial state
- **Testing**: assert that actions produce expected states without a CRDT
- **Server-side rendering**: maintain state without a full CRDT runtime
- **Read-your-writes in batches**: drafts use `step` to overlay pending
  actions on the source state (see §8.2)

### 4.6. Ref methods as action constructors

The developer writes imperative code. Under the hood, ref methods
construct actions and dispatch them through the context:

```
ref.insert(3, "Hello")
→ dispatches TextAction { type: "text", ops: [{ retain: 3 }, { insert: "Hello" }] }

ref.set("darkMode", true)
→ dispatches MapAction { type: "map", set: { darkMode: true } }

ref.push(item)
→ dispatches ListAction { type: "list", ops: [{ retain: length }, { insert: [item] }] }
```

The ref is a thin shell: path resolution + action construction. The
context provides the executor (see §8).

### 4.7. Compiler integration

The Kinetic compiler uses the action type to determine patching strategy.
For **built-in** action types, it emits optimized direct calls at build
time:

```typescript
// Compiler output for TextAction:
_feed.subscribe((action) => {
  applyTextPatch(_p, action)  // direct call, no dispatch
})
```

For **unknown** action types, it emits a call through a runtime dispatch
table with a replace fallback:

```typescript
// Compiler output for unknown action type:
_feed.subscribe((action) => {
  applyAction(_p, _feed, action)  // runtime dispatch
})
```

Third-party adapters register custom patch strategies **at module load
time** (after import, before first render — normal module initialization
order):

```typescript
import { registerPatchStrategy } from "@loro-extended/kinetic/runtime"

registerPatchStrategy("richtext", (element, feed, action) => {
  // Custom DOM patching for rich text actions
})
```

This gives zero overhead for built-in actions (direct calls), extensibility
for third-party actions (runtime registration), and a correct fallback
(replace) if no strategy is registered.

## 5. Feeds: The Unified Reactive Protocol

A **feed** is a reactive value with a **head** (the current state) and a
**tail** (a stream of actions describing future changes). You read the
head to see what's there now; you subscribe to the tail to learn what
changes next.

> *Theoretical note.* This is a coalgebra — the dual of an algebra.
> In automata theory the structure is called a Moore machine
> (`S → Output × (Input → S)`). Our encoding replaces the transition
> function with a push-based action stream, making it efficient for
> reactive UI (O(k) patching where k is the change size).

The feed protocol is expressed through a single symbol:

```typescript
const FEED = Symbol.for("kinetic:feed")

interface Feed<S, A extends ActionBase> {
  /** The head — current value, always live (a getter) */
  readonly head: S
  /** The tail — subscribe to actions as they flow */
  subscribe(callback: (action: A) => void): () => void
}

interface Feedable<S, A extends ActionBase> {
  readonly [FEED]: Feed<S, A>
}
```

### 5.1. One symbol, one concept

The previous design used two symbols (`[SNAPSHOT]` and `[REACTIVE]`).
The unified design collapses them: `[FEED]` means "this value is a
feed — it has a head you can read and a tail you can subscribe to."

Properties of the unified design:

- **Head and subscribe are co-located.** You can't subscribe without
  being able to read, or read without being able to subscribe.
- **Static data sources still work.** A non-reactive source returns
  `{ head: value, subscribe: () => () => {} }` — a feed whose tail
  never emits.
- **No `self` parameter.** The `[FEED]` getter has `this` naturally.

### 5.2. Caching with WeakMap

The `[FEED]` getter is defined once on the prototype. A module-scoped
`WeakMap` caches the feed object per instance, avoiding per-instance
field allocation while preserving referential identity:

```typescript
const feeds = new WeakMap<object, Feed<any, any>>()

export function getOrCreateFeed<S, A extends ActionBase>(
  ref: object,
  factory: () => Feed<S, A>,
): Feed<S, A> {
  let feed = feeds.get(ref) as Feed<S, A> | undefined
  if (!feed) {
    feed = factory()
    feeds.set(ref, feed)
  }
  return feed
}

// On any ref class:
get [FEED](): Feed<S, A> {
  return getOrCreateFeed(this, () => ({
    get head() { return readCurrentValue(self) },
    subscribe: (cb) => subscribeToChanges(self, cb),
  }))
}
```

Properties: no per-instance allocation at construction time; feed
created lazily on first access; referential identity (`ref[FEED] ===
ref[FEED]`); GC-safe (WeakMap entry disappears when ref is collected);
no `self`-casting boilerplate.

### 5.3. The Feed interpreter

The feed is an interpreter — it produces a `Feed` at each schema node:

```
feed.text = (ctx, path) => ({
  get head() { return getText(ctx, path).toString() },
  subscribe(cb) { return getText(ctx, path).subscribe(translateText(cb)) },
})

feed.counter = (ctx, path) => ({
  get head() { return getCounter(ctx, path).value },
  subscribe(cb) { return getCounter(ctx, path).subscribe(translateCounter(cb)) },
})

feed.struct = (ctx, path, fields) => ({
  get head() { return mapValues(fields, f => f().head) },
  subscribe(cb) { return getMap(ctx, path).subscribe(translateMap(cb)) },
})

feed.list = (ctx, path, item) => ({
  get head() { return self },
  subscribe(cb) { return getList(ctx, path).subscribe(translateList(cb)) },
})
```

The feed interpreter is **backend-specific** in its leaf implementations
(how to read, how to subscribe) but **structurally uniform** (the walker
is the same for every backend). A Firestore backend provides different
leaf implementations but uses the same walker.

### 5.4. Capability decomposition

A feed provides **observation** (read + subscribe). Mutation is a
separate, orthogonal capability provided by the Writable interpreter.
Today's `TypedRef` combines both:

| Capability | What it provides | Protocol |
|---|---|---|
| **Readable** | Current value observation | `feed.head` |
| **Subscribable** | Action notification | `feed.subscribe` |
| **Writable** | State mutation | `.set()`, `.insert()`, etc. |

A feed is Readable + Subscribable. The Kinetic compiler only needs feeds
— it reads heads and subscribes to tails but never mutates. The
`change()` blocks need the Writable capability. `toJSON()` only needs
the Readable capability.

## 6. Zeros: Separated from the Schema

A **zero** is the identity element for the `step` monoid — the state
before any actions have been applied. Today, zeros (placeholders) are
baked into shape definitions via `_placeholder`. We separate them.

### 6.1. Why separate?

1. **Different contexts need different zeros.** A "new document" zero
   might have `title: "Untitled"`. A "loading skeleton" zero might have
   `title: ""`. A "test fixture" zero might have `title: "Test Document"`.

2. **Zero is just an interpreter.** The schema is a functor; zero is a
   natural transformation from the schema to plain values. Baking it into
   the schema is like baking a function into its domain type.

3. **It simplifies the Shape interface.** Without the placeholder, shapes
   carry only structural information. No `_placeholder` phantom type.

### 6.2. Structural zero

Every schema has a mechanical **structural zero** — the type-level
default that requires no user configuration:

```
structural.text    = ""
structural.counter = 0
structural.value   = typeDefault(shape)   -- "", 0, false, null, etc.
structural.list    = []
structural.struct  = mapValues(shapes, deriveStructural)
structural.record  = {}
structural.tree    = []
structural.doc     = mapValues(shapes, deriveStructural)
```

This is `derivePlaceholder` today, but without consulting `_placeholder`.

### 6.3. Custom zeros

Custom zeros are defined separately, typed against the schema:

```typescript
const newDocZero = Zero.for(ChatSchema, {
  title: "Untitled",
  count: 0,
  messages: [],
})

// Type-checked: must match Plain<typeof ChatSchema>
```

### 6.4. Partial zeros and overlay

Zeros compose via **overlay** — pointwise `firstDefined` lifted through
the schema structure:

```typescript
const partial   = Zero.partial(ChatSchema, { title: "Untitled" })
const base      = Zero.structural(ChatSchema)
const effective = Zero.overlay(partial, base)
// { title: "Untitled", count: 0, messages: [] }
```

This generalizes the current overlay system. The fallback chain becomes:

```
1. CRDT state            (truth from the backend)
2. Optimistic state      (local changes not yet confirmed)
3. Custom zero           (user-configured defaults)
4. Structural zero       (type-level defaults)
```

Each level produces a candidate value. The merge rule is `firstDefined`
at each leaf — the monoid operation on `Maybe<T>` with "first wins."

Today's `overlayPlaceholder` implements levels 1 + 3+4 (merged). The
new design makes each level explicit and composable.

### 6.5. The overlay interpreter

The overlay is itself an interpreter that combines two `Plain`
interpretations. It's **not** a simple `??` fallback — it's a deep
structural merge that needs shape information at each node (structs
merge per-key, lists merge per-item, discriminated unions match on
discriminant before merging):

```
overlay(primary, fallback).text    = primary ?? fallback
overlay(primary, fallback).counter = primary ?? fallback
overlay(primary, fallback).list    = primary?.map((item, i) =>
  overlay(item, fallback[i] ?? itemZero)) ?? fallback
overlay(primary, fallback).struct  = mergeKeys(primary, fallback, perKeyOverlay)
```

## 7. Interpreter Composition

Interpreters compose. Given interpreters `F` and `G`, we can build:

### 7.1. Product interpreter (genuinely independent interpretations)

```typescript
product(F, G).text = (ctx, path) => [F.text(ctx, path), G.text(ctx, path)]
product(F, G).struct = (ctx, path, pairedFields) => {
  const fieldsA = mapValues(pairedFields, ([a, _]) => a)
  const fieldsB = mapValues(pairedFields, ([_, b]) => b)
  return [F.struct(ctx, path, fieldsA), G.struct(ctx, path, fieldsB)]
}
```

Product works for **any pair with the same Ctx type**. At struct/list/doc
nodes, paired results must be unzipped before passing to each interpreter.

Product is appropriate for genuinely independent interpretations, e.g.
`product(plain, path)` — you want both a current value and a path
selector, computed independently.

### 7.2. Enrich combinator (adding protocol to an interpretation)

```typescript
enrich<A, Protocol>(
  base: Interpreter<Ctx, A>,
  protocol: (result: A, ctx: Ctx, path: Path) => Protocol,
): Interpreter<Ctx, A & Protocol>
```

Enrich takes an interpretation and **decorates** each result with
additional protocol. This is more appropriate than `product` when
the added protocol depends on the base result.

Today's `TypedRef` is `enrich(writable, withFeed)` — the writable
ref is the state surface, and the feed's `head` reads from it:

```typescript
const withFeed = (writableRef, ctx, path) => ({
  [FEED]: getOrCreateFeed(writableRef, () => ({
    get head() { return readFrom(writableRef) },
    subscribe(cb) { return subscribeAt(ctx, path, cb) },
  }))
})
```

The enrichment says: "take this writable thing and make it observable."

### 7.3. Overlay combinator (structural merge with fallback)

```typescript
overlay(primary, fallback).text = (ctx, path) =>
  primary.text(ctx, path) ?? fallback.text(ctx, path)
overlay(primary, fallback).struct = (ctx, path, fields) =>
  mergeStructFields(primary.struct(...), fallback.struct(...))
```

This is **not** a simple `??` at the top level — it's a deep structural
merge that recurses through the schema. The overlay at each node is
shape-aware: structs merge per-key, lists merge per-item, discriminated
unions match on discriminant before merging.

This is how the CRDT state + zero system works:
`overlay(crdtInterpreter, zeroInterpreter)`.

## 8. Context, Dispatch, and Namespace

Three properties of the current ref system must be preserved by the
interpreter algebra. They are not new interpreters or combinators —
they are **structural constraints** on `Ctx` and `A`.

### 8.1. Context accumulation (upward references)

A `PlainRef` for `doc.settings.darkMode` doesn't own a Loro container —
there is no `LoroBool`. It holds a reference to its **parent** container
(the `LoroMap` for `settings`) plus a key (`"darkMode"`). When you call
`.set(true)`, it reaches up to the parent map.

This means the context isn't uniform — it **accumulates** as the walker
descends the tree:

```
doc level:        Ctx = { doc, autoCommit }
struct level:     Ctx = { doc, autoCommit, container: LoroMap, path }
value level:      Ctx = { doc, autoCommit, container: LoroMap, path, key }
```

At each level the context is enriched with the current node's
information. In attribute grammar terms, these are **inherited
attributes** flowing from parent to child.

The interpreter's `struct` case returns not only its result `A` but
implicitly defines how child contexts are derived — today this is
`buildChildTypedRefParams`, which computes `getContainer`, `pathPrefix`,
`placeholder`, and `mergeable` for each child.

The critical property: **a ref is portable.** You can detach it from the
tree and pass it to a function. It carries its context (parent container,
doc reference, commit policy) as a closure. The function calls `.get()`
and `.set()` without knowing anything about the doc, the backend, or the
transaction state:

```typescript
function clamp(ref: PlainRef<number>, min: number, max: number) {
  const v = ref.get()
  if (v < min) ref.set(min)
  if (v > max) ref.set(max)
}

clamp(doc.settings.volume, 0, 100)   // works
clamp(doc2.settings.volume, 0, 50)   // works — different doc's context
```

### 8.2. Transparent batching (commit policy as context)

Whether you're inside `change()` or not, the same `.set()` call works.
The difference is in the context:

| | Outside `change()` | Inside `change()` |
|---|---|---|
| `autoCommit` | `true` | `false` |
| `batchedMutation` | `false` | `true` |
| Commit timing | After each write | At block boundary |

This is a **transaction** in the algebraic sense. The context carries the
commit policy. Mutations go through the context, which determines when
they're committed. The interpreter doesn't know or care.

Formally, the context is monadic — mutations are effects, and the context
determines how effects are sequenced and committed. Auto-commit is the
identity monad (each effect is immediately realized). Batched mode is a
writer monad (effects accumulate, then are flushed).

The key devX property is **transparency**: a function that operates on
refs doesn't need to know about transactions:

```typescript
function resetSettings(settings: StructRef<SettingsShape>) {
  settings.darkMode.set(false)
  settings.fontSize.set(14)
}

// Works outside change() — two auto-commits:
resetSettings(doc.settings)

// Works inside change() — zero commits (batched):
change(doc, draft => { resetSettings(draft.settings) })
```

**Read-your-writes in batches.** Inside `change()`, reads must reflect
pending writes. Today this works because mutations execute immediately
against the CRDT (just uncommitted). In a future architecture with
reified actions, `step` provides the mechanism: the draft overlays
pending actions on the source state via `step(source, pendingActions)`.
Reads go through the overlay. Writes append to the pending list. Commit
flushes to the backend.

### 8.3. Namespace isolation (symbol-keyed protocol)

A `StructRef` for `{ subscribe: boolean, title: string }` must let you
write `ref.subscribe` to access the boolean, not some internal method.
The entire string-key namespace belongs to the schema.

Internal protocol lives behind well-known symbols:

```
// String namespace — schema properties (developer's domain):
ref.subscribe           → PlainRef<boolean>
ref.title               → PlainRef<string>

// Symbol namespace — protocol (hidden machinery):
ref[FEED]               → Feed<StructRef, MapAction>
ref[LORO_SYMBOL]        → LoroMap (escape hatch to native Loro)
ref[INTERNAL_SYMBOL]    → StructRefInternals
```

This is a constraint on the `enrich` combinator: enrichments add only
**symbol-keyed** properties. The string namespace is never polluted.
This is enforceable in `enrich`'s type signature.

Consequences:

- `Object.keys(ref)` returns only schema property names. React, JSON
  serialization, and `for...in` see only the developer's data.
- The `[FEED]` symbol is non-enumerable by construction.
- The `loro()` and `ext()` accessor functions are sugar over symbol
  lookup — they provide typed access to the escape hatches without
  adding string-keyed methods.
- `toJSON()` is currently string-keyed (a historical exception). The
  principle says it should eventually move to `ext(ref).toJSON()` or a
  symbol, freeing the string `"toJSON"` for use as a schema property.

## 9. What This Means for the Codebase

### 9.1. Backend independence

The schema + interpreter separation means the feed protocol is not
Loro-specific. A backend must provide:

1. **A context type** (e.g., `LoroDoc`, or a Firestore reference)
2. **Leaf interpreter implementations** for read, subscribe, and write
3. **An action translation layer** mapping backend events to `ActionBase`
   subtypes

Everything else — the tree walking, proxy creation, type inference,
compiler detection, runtime subscription — works unchanged.

### 9.2. Eliminating parallel implementations

Today's codebase has at least 10 separate dispatch sites on
`shape._type`. With the interpreter pattern, each becomes a small
interpreter definition passed to a single generic `interpret()` function.
The Proxy machinery, lazy property creation, and caching live in one
place.

### 9.3. New interpretations become trivial

Want a "validation" interpreter that checks invariants at each node?
A "migration" interpreter that transforms shapes? A "diff" interpreter
that computes structural differences? Each is a small `Interpreter<Ctx, A>`
definition — no new proxy system, no new tree walker.

### 9.4. The list item problem dissolves

The question "should `for (const item of doc.todos)` yield a string or a
PlainRef?" becomes: "which interpreter is active?" In a reactive context
(the Kinetic compiler's generated code), the feed interpreter is active,
so items are feeds. In a plain context (`toJSON`), the Plain interpreter
is active, so items are strings. The schema is the same; the
interpretation differs.

## 10. Type-Level Encoding

TypeScript's type system can express the interpreter pattern via
mapped types and conditional types:

```typescript
// The schema is the syntax
type Schema =
  | { _type: "text" }
  | { _type: "counter" }
  | { _type: "value"; valueType: string }
  | { _type: "list"; shape: Schema }
  | { _type: "struct"; shapes: Record<string, Schema> }
  | { _type: "doc"; shapes: Record<string, Schema> }
  // ...

// An interpretation is a type-level function from Schema to types
type Interpret<S extends Schema, Mode extends string> =
  S extends { _type: "text" }
    ? Mode extends "plain" ? string
    : Mode extends "writable" ? TextRef
    : Mode extends "feed" ? Feed<string, TextAction>
    : Mode extends "path" ? PathSelector<string>
    : never
  : S extends { _type: "list"; shape: infer Item extends Schema }
    ? Mode extends "plain" ? Interpret<Item, Mode>[]
    : Mode extends "writable" ? ListRef<Item>
    : Mode extends "feed" ? Feed<ListRef<Item>, ListAction<Interpret<Item, "plain">>>
    : never
  : // ...
```

### 10.1. Preserving the phantom type advantage

The current approach where each shape carries `_plain`, `_mutable`, etc.
has a valuable property: the type system knows the interpretation result
at the **definition site** via simple indexed access (`S["_mutable"]`),
not a recursive conditional type chain. This gives fast type-checking.

A hybrid approach preserves this: shapes carry phantom types for known
modes (fast path), and the `Interpret<S, Mode>` pattern handles extension
to new modes. Module augmentation enables third-party mode registration
without modifying the schema package.

## 11. Relationship to Existing Code

This document describes the *target architecture*. The path from here
to there is incremental:

### Phase 0 (current state)
- Shapes carry four phantom types including `_placeholder`
- Each interpretation is a separate `switch (shape._type)` implementation
- Reactive protocol is split across two symbols (`[SNAPSHOT]` + `[REACTIVE]`)
- Delta vocabulary is a closed union (`ReactiveDelta`)
- Operations (developer → backend) and deltas (backend → observer) are
  separate, unrelated types
- Mutable/Draft distinction is a type-level phantom with no runtime
  behavioral difference (`resolveValueForBatchedMutation` is identical
  to `createPlainValueRefForProperty`)

### Spike: Validate primitives in isolation
- New `packages/schema/` package with zero dependencies on existing code
- Implement `SchemaF` node types, `ActionBase` + concrete action types,
  `Feed<S, A>` with `[FEED]` symbol and WeakMap caching
- Implement `step` per action type, `Zero.structural`, `Zero.overlay`
- Implement `Interpreter<Ctx, A>`, `interpret()` catamorphism
- Implement `enrich`, `product`, `overlay` combinators
- Test round-trips: `step(Zero.structural(schema), action) → expected`
- See `.plans/schema-algebra-spike.md` for the full plan

### Phase 1: Unify the feed protocol
- Collapse `[SNAPSHOT]` + `[REACTIVE]` into a single `[FEED]` that
  returns a `Feed<S, A>` object
- Implement `WeakMap`-based caching via `getOrCreateFeed`
- Remove the `self` parameter pattern (getter has `this` naturally)
- `Feed` interface: `{ head: S, subscribe(cb) → unsub }`
- Open the action vocabulary: `ActionBase` protocol with string
  discriminant
- Update Kinetic compiler's `isReactiveType` / `getDeltaKind` to detect
  `[FEED]` and extract `A` from `Feed<S, A>`
- Update Kinetic runtime to consume `feed.head` / `feed.subscribe`
- No external API shape changes; `TypedDoc`, `TypedRef` types remain
  stable

### Phase 2: Separate zeros from schema
- Remove `_placeholder` from shape definitions
- Define `Zero.structural(schema)` — mechanical type-level defaults
- Define `Zero.for(schema, value)` — type-checked custom zeros
- Define `Zero.partial(schema, partialValue)` + `Zero.overlay`
  composition
- Refactor `derivePlaceholder` → `Zero.structural`
- Refactor `overlayPlaceholder` → `Zero.overlay`
- `createTypedDoc` accepts a zero argument instead of reading
  `_placeholder`

### Phase 3: Unify actions
- Define `ActionBase` and concrete action types in
  `@loro-extended/reactive`
- Refactor `reactive-bridge.ts` to produce actions instead of
  `ReactiveDelta`
- Ref mutation methods construct and dispatch actions through context
- Implement `step` for each action type
- Feed subscriptions deliver actions; notification projection is opt-in

### Phase 4: Extract the generic walker
- Factor out the recursive tree-walking from all 10+ dispatch sites
- Define the `Interpreter<Ctx, A>` interface with thunk-based fields
- Implement `interpret(schema, interpreter, ctx)` — the catamorphism
- Rewrite existing implementations as interpreter instances
- No API changes; internal refactor only

### Phase 5: Decouple schema from Loro
- Move shape definitions and the walker to `@loro-extended/schema`
- `@loro-extended/change` provides the Loro-specific interpreters
- `@loro-extended/reactive` provides `Feed`, `ActionBase`, built-in
  action types, `getOrCreateFeed`
- Composition combinators (`enrich`, `product`, `overlay`) live in
  `@loro-extended/schema`

### Phase 6: Open the interpretation parameter
- Replace phantom type tuple with an open `Interpret<S, Mode>` mapped
  type
- Preserve phantom types for built-in modes (fast type-checking)
- Use module augmentation for third-party mode registration
- Enable third-party interpreters without modifying the schema package

## 12. Composable Primitives Summary

| Primitive | Type | Purpose |
|---|---|---|
| **Schema** | `Fix(SchemaF)` | Pure structure description, no behavior, no defaults |
| **Interpreter** | `SchemaF<A> → A` | What to do at each node |
| **interpret** | `(Schema, Interpreter<Ctx, A>, Ctx) → A` | The catamorphism |
| **Action** | `{ type: string, ... }` | Universal currency of change (in and out) |
| **Feed** | `{ head: S, subscribe(A → void) → unsub }` | Unified reactive observation |
| **step** | `(State, Action) → State` | Pure state transition, no backend needed |
| **Zero** | `Interpreter<void, Plain<S>>` | Default values, separated from schema |
| **overlay** | `(Partial<Plain<S>>, Plain<S>) → Plain<S>` | Structural merge with fallback |
| **enrich** | `(Interp<A>, decorator) → Interp<A & Protocol>` | Add protocol to interpretation |
| **product** | `(Interp<A>, Interp<B>) → Interp<[A, B]>` | Parallel independent composition |
| **notify** | `Action → Notification` | Lossy projection (optional optimization) |
| **execute** | `(Ctx, Path, Action) → void` | Backend-specific action execution |

## 13. Notation Reference

| Term | Meaning |
|---|---|
| **F-algebra** | A function `F<A> → A` that collapses one layer of functor `F` into result `A` |
| **Catamorphism** (`cata`) | The unique fold over a recursive type — our `interpret()` function |
| **Coalgebra** | The dual: `A → F<A>` — unfolds behavior from state |
| **Feed** | A coalgebra with push-based actions — `{ head: S, subscribe: (A → void) → unsub }` |
| **Action** | A description of change — flows developer → backend and backend → observer |
| **Natural transformation** | A structure-preserving map between functors — each interpreter row in §1 |
| **Functor** | A structure-preserving map between categories — `SchemaF` maps types to types |
| **Initial algebra** | The "free" recursive type — `Fix(SchemaF)` is the schema itself |
| **Product** | Parallel composition — two interpretations run independently |
| **Enrich** | Decoration — add symbol-keyed protocol to an existing interpretation |
| **Zero** | Identity element of the step monoid — state before any actions |

## 14. Learnings from Codebase Audit

These observations came from a thorough audit of the implemented code and
informed the design above.

### 14.1. True dispatch site count

The original document claimed 7 parallel implementations. Actual count
is **10-11** independent `shape._type` dispatch sites across
`derive-placeholder.ts`, `overlay.ts`, `path-builder.ts`, `conversion.ts`,
`typed-doc.ts` (reconstructFromFlattened), `typed-refs/utils.ts`
(createContainerTypedRef, assignPlainValueToTypedRef,
unwrapReadonlyPrimitive), `MapBasedRefInternals.getOrCreateRef`,
`DocRefInternals.getOrCreateTypedRef`,
`ListRefBaseInternals.getMutableItem`, plus per-class `toJSON()`.

### 14.2. Mutable/Draft distinction is phantom-only

`resolveValueForBatchedMutation` is literally `return
createPlainValueRefForProperty(...)`. The Draft mode uses the same ref
classes with a boolean flag (`batchedMutation`). The Mutable/Draft
"contextual interpreter" from the original §5.3 doesn't exist at runtime.
This is good news — it means draft is a type-level concern only.

### 14.3. Reactivity is per-ref, not a separate walk

The original document framed the reactive interpreter as a separate tree
walk. In reality, every `TypedRef` inherits `[REACTIVE]` and `[SNAPSHOT]`
from the base class. There is no reactive tree walk — each ref
independently implements the protocol. This is closer to
`enrich(writable, withFeed)` than a separate feed interpreter.

### 14.4. Overlay is a deep structural merge

The overlay system (`overlayPlaceholder` / `mergeValue`) is not a simple
`??` fallback. It's a recursive shape-aware merge: structs merge per-key,
lists merge per-item with derived item placeholders, discriminated unions
match on discriminant before merging. The overlay combinator must be
shape-aware, not just value-aware.

### 14.5. Laziness is deeply embedded

`DocRefInternals.propertyCache`, `MapBasedRefInternals.refCache`,
`ListRefBaseInternals.itemCache` — the codebase is committed to
cache-on-first-access. The interpreter's struct/doc fields must be
thunks (not pre-computed values) to preserve this. Eagerly interpreting
all fields would be a performance regression for large document trees.

### 14.6. Mergeable storage is orthogonal to interpretation

The `mergeable` flag changes how containers are stored (flattened root
storage vs nested hierarchical), creating an entire tree walk
(`reconstructFromFlattened`) that only exists for this storage strategy.
Storage strategy should be a parameter on `Ctx`, not a concern of the
interpreter. The `TypedRefParams` already carries `mergeable`,
`pathPrefix`, and `getContainer` — these are "how to access storage"
parameters orthogonal to "what to do at each node."

### 14.7. The `@loro-extended/schema` package is empty

`packages/schema/` contains only this theory document. No `package.json`,
no source code. All shape definitions live in `@loro-extended/change`
at `packages/change/src/shape.ts`. The shape system is tightly coupled
to Loro-specific ref types through the phantom types (`_mutable: TextRef`,
etc.). Phase 5 (decouple schema from Loro) has not started.

### 14.8. Operations and deltas are the same information

The `reactive-bridge.ts` translates Loro diffs into `ReactiveDelta`.
Ref methods like `TextRef.insert(0, "hi")` construct the same structural
information (retain/insert/delete) but call Loro APIs directly. These are
two encodings of the same semantic content — one going in (operations),
one coming out (deltas). Unifying them into `Action` eliminates the
separate vocabularies and enables `step` as the pure bridge between them.

### 14.9. Full mutation reification was considered and rejected

We explored making every `ref.set()` / `ref.insert()` produce data
(the free monad pattern) instead of executing immediately. This would
enable pure mutation capture, undo/redo as program inversion, and
multi-backend replay. However, read-your-writes (`const size =
draft.settings.fontSize.get()` immediately after `.set(16)`) is deeply
relied upon in existing `change()` blocks. The free monad requires either
a monadic DSL (hostile devX), a shadow state (complexity), or a compiler
transform. **Rejected** in favor of: mutations execute immediately, but
the Action type unifies the vocabulary so the same structure flows in
both directions. `step` provides read-your-writes in future batch
architectures by overlaying pending actions on source state.

## 15. Open Questions

1. **Thunk forcing and caching in the walker.** The interpreter receives
   thunks for struct/doc fields. Should the walker provide a caching
   layer (force-once semantics), or should interpreters manage their own
   caches? The current `refCache` pattern suggests force-once is
   desirable.

2. **Proxy vs plain object.** `StructRef` and `TypedDoc` use Proxies for
   dynamic property dispatch. `ListRef` and `TextRef` are plain classes.
   Can the walker unify these, or is the Proxy/class distinction
   inherent to the node kind?

3. **Incremental adoption.** Can we introduce the unified feed protocol
   behind the existing API surface, so no external consumers break? The
   `TypedDoc`, `TypedRef`, and `PlainRef` types should remain stable;
   only the internal protocol changes.

4. **Zero devX.** Separating zeros from schemas is cleaner but less
   convenient than the co-located `.placeholder()` method. Should
   `createTypedDoc` accept a zero as an option, or should it derive
   from a builder pattern?

5. **Storage strategy abstraction.** Should the `Ctx` type encode storage
   strategy (mergeable/hierarchical), or should there be a separate
   "storage adapter" composed with the interpreter? The current
   `TypedRefParams` bundles both concerns.

6. **Action granularity for maps.** `MapAction` carries `set` and
   `delete` as full key-value pairs. Should it also support a
   cursor-based encoding for ordered maps, or is key-value sufficient
   for all map-backed schema nodes (struct, record, doc)?

7. **`toJSON` migration.** Today `toJSON` is a string-keyed method,
   violating namespace isolation (a schema property named `"toJSON"` would
   collide). Moving it to `ext(ref).toJSON()` or a symbol is a breaking
   change. When and how to migrate?
